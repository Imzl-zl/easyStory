# easyStory 文档总览

easyStory 是一个面向中文小说创作场景的 AI 创作系统。它的目标不是替代作者做“一键流水线出文”，而是帮助用户用更低门槛的方式完成从题材设定、故事大纲、章节规划、正文生成、内容审核到最终导出的完整创作闭环。

这套文档的作用，是把四件事情说清楚：

1. 这个系统到底是做什么的。
2. 为什么要这样做，而不是做一个普通的聊天框或纯自动流水线。
3. 这个系统要满足哪些用户和哪些需求。
4. 为什么最终选择这一套技术栈，以及它如何适配大多数用户。

## 1. 系统是什么

easyStory 是一个“节点式 AI 小说创作工作台”。

它把小说创作拆成一组可管理、可审核、可精修的节点：

- 题材设定
- 故事大纲
- 章节规划
- 章节生成
- 内容审核
- 局部精修
- 导出成稿

每个节点都不是一次性黑盒输出，而是一个可以暂停、查看、确认、修改、重试和回退的创作步骤。

换句话说，easyStory 不是“给一个 Prompt 然后等全文吐出来”，而是“把长篇创作拆成多步协作，让 AI 负责生成，人负责把关”。

## 2. 为什么要做这个系统

当前大多数 AI 写作方式有三个明显问题：

### 2.1 普通聊天框太自由，但不稳定

- 直接用 ChatGPT、Claude 或其他大模型写小说，通常起步快，但结果不稳定。
- 内容容易前后不一致，人物和设定容易漂移。
- 当用户想修改局部内容时，往往只能重新提一轮 Prompt，成本很高。

### 2.2 纯自动流水线效率高，但质量难控

- 一键批量生成可以快速出结果，但结果通常不可控。
- 用户很难在关键步骤中插入判断和修改。
- 一旦某一步偏了，后面全部内容都会跟着跑偏。

### 2.3 现有专业工具要么太重，要么太窄

- 有些工具偏工作流平台，灵活但门槛高。
- 有些工具偏模板站，容易上手但难以深度控制。
- 有些工具偏英文写作场景，对中文网文和长篇连载支持不够友好。

因此，easyStory 想解决的不是“能不能生成文本”，而是下面这些更实际的问题：

- 怎么让普通用户也能比较稳定地写出一篇完整故事。
- 怎么让用户在关键节点保留判断权，而不是被自动流程绑架。
- 怎么把提示词、模型、参数和流程沉淀成可复用配置，而不是每次从头摸索。
- 怎么让新手能开箱即用，同时让进阶用户还能继续调优。

## 3. 这个系统适合谁

easyStory 不是只给提示词高手用，也不是只给工作流工程师用。它需要覆盖大多数真正会用 AI 写作的人。

### 3.1 主要用户

| 用户类型 | 当前问题 | easyStory 解决方式 |
|---|---|---|
| 新手作者 | 不会写复杂 Prompt，不知道怎么起稿 | 提供默认模板和默认流程，先跑通第一篇 |
| 普通创作者 | 会写一些提示词，但流程容易乱 | 把创作拆成节点，降低上下文失控 |
| 职业作者 / 工作室 | 需要复用经验、稳定出稿 | 沉淀模板、节点配置、模型档案和版本记录 |
| 高阶用户 | 想自定义提示词、流程和模型参数 | 在系统内做配置，而不是每次手工拼装 |

### 3.2 不以谁为中心

本系统第一阶段不以以下人群为中心：

- 需要多人实时协作的大型团队。
- 需要开放插件市场和第三方执行环境的平台型用户。
- 需要大规模知识检索、企业级权限和多租户隔离的 SaaS 运营团队。

这些人群不是不重要，而是不应该主导第一阶段产品设计。

## 4. 面向大多数用户的设计原则

为了适配大多数用户，系统设计必须遵守下面几条原则：

### 4.1 开箱即用

- 第一次使用时，不要求用户先理解工作流引擎、插件机制、MCP 或高级参数。
- 默认提供一条可以直接跑通的创作流程。
- 默认模板应覆盖最常见的中文创作场景，例如网文、短篇、中篇。

### 4.2 渐进式复杂度

- 新手先用默认配置。
- 熟练后可以调整模型、提示词和节点参数。
- 高阶用户再进入自定义模板和高级工作流。

系统不能一上来把所有高级能力都摆到用户面前。

### 4.3 人机协作，而不是全自动替代

- 生成由 AI 负责。
- 审核和方向判断由人负责。
- 每步可通过、精修、重试、回退。

这能同时兼顾效率和质量。

### 4.4 Web 优先，跨环境可用

- 用户应尽量通过浏览器使用系统，不要求安装复杂客户端。
- 开发和部署方式要尽量兼容普通开发者、自托管用户和小团队。

### 4.5 兼容主流模型生态

- 系统不应把用户锁死在单一模型提供商。
- 至少要兼容 OpenAI 兼容接口、Anthropic、Google Gemini 以及本地模型扩展路径。

## 5. 系统核心能力概览

### 5.1 用户视角的能力

- 创建创作项目
- 选择模板或默认流程
- 生成大纲
- 生成章节规划
- 逐章生成正文
- 审核和精修内容
- 追踪版本变化
- 导出最终稿

### 5.2 系统视角的能力

- 持久化项目、节点运行和内容版本
- 管理模型配置和供应商调用
- 记录审核动作和生成日志
- 恢复中断的节点状态
- 为后续模板化和平台化保留扩展点

## 6. 第一阶段做什么，不做什么

### 6.1 第一阶段必须做

- 默认创作流程
- 审核动作闭环
- 内容版本留存
- 模型档案配置
- Markdown 导出
- 单机可运行与单机可部署

### 6.2 第一阶段不强求

- 插件市场
- 模板市场
- 向量检索
- MCP Server
- Redis 和复杂后台任务系统
- 团队协作
- Kubernetes

## 7. 技术路线一句话说明

easyStory 第一阶段选择的是一条“足够成熟、足够主流、足够容易落地”的技术路线：

- 前端用 `Next.js + React` 搭建 Web 工作台
- 后端用 `FastAPI + Pydantic + SQLAlchemy` 提供 API 和业务编排
- 工作流用 `LangGraph` 承载节点状态和流程控制
- 模型接入用 `LiteLLM` 统一不同供应商接口
- 开发环境用 `SQLite` 快速启动，部署默认用 `PostgreSQL`

这样做的原因很直接：

- 对开发者足够常见，学习和招聘成本低。
- 对用户足够稳妥，容易跑起来。
- 对后续演进足够友好，不至于一开始就做成过重的平台。

## 9. 当前结论

easyStory 的核心方向已经明确：

- 它是一个节点式 AI 小说创作系统。
- 它强调人机协作，不强调全自动流水线。
- 它要适配大多数真正写作的人，而不是只服务少数工作流高手。
- 它的第一阶段目标是把“默认流程跑通 + 可审核 + 可导出”这条最小闭环做好。
