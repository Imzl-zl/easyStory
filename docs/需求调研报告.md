## 1. 文档目标

本文件回答六个问题：

1. easyStory 到底是什么系统。
2. 为什么要做这样一个系统。
3. 这个系统主要服务哪些用户。
4. 它需要满足哪些核心需求。
5. 第一阶段必须完成什么，明确不做什么。
6. 它如何适配大多数用户，而不是只适配少数高手。

## 2. 系统定义

### 2.1 产品定位

easyStory 是一个面向中文小说创作场景的 AI 创作系统。它以“节点式创作工作台”为核心交互形态，把完整写作过程拆成多个可以生成、审核、精修和回退的步骤。

它不是以下三种东西：

- 不是普通聊天机器人。
- 不是纯自动流水线批量出文工具。
- 不是第一阶段就追求开放插件生态的平台。

它首先是一个帮助用户稳定完成创作闭环的工作系统。

### 2.2 一句话定义

easyStory = 面向中文小说创作的 AI 工作台 + 节点式流程 + 人工审核闭环 + 可复用配置体系。

### 2.3 核心价值

| 价值点 | 说明 |
|---|---|
| 稳定性 | 把长文本创作拆成节点，降低一次性生成失控的概率 |
| 可控性 | 每个关键步骤都允许人工审核和干预 |
| 可复用性 | 把模板、提示词、模型和参数沉淀成可复用配置 |
| 可成长性 | 新手可直接用默认流程，进阶用户可逐步深入配置 |

## 3. 为什么要做

### 3.1 当前创作方式的痛点

#### 直接用通用大模型写小说的问题

- 容易出现前后设定不一致。
- 长文本上下文容易漂移。
- 局部修改代价高，通常需要重新提示。
- 用户不知道哪一步出了问题，只能整体重来。

#### 纯自动化流水线的问题

- 效率高，但创作质量不稳定。
- 用户缺少在中间步骤插手的机会。
- 一旦前面节点跑偏，后面节点会持续放大偏差。

#### 现有工具的问题

- 有些工具偏通用工作流平台，对创作者来说门槛太高。
- 有些工具偏固定模板工具，灵活性不够。
- 有些工具偏英文内容创作，对中文网文、长篇连载支持不理想。

### 3.2 这个系统要解决的核心问题

easyStory 不只是解决“能否生成文本”，而是解决下面这些问题：

- 如何让普通用户在较少 Prompt 技巧的前提下也能开始写作。
- 如何让内容生成过程可拆解、可回看、可修正。
- 如何把“模型 + 提示词 + 参数 + 流程”沉淀为资产。
- 如何在效率和质量之间找到平衡。

## 4. 目标用户

### 4.1 用户分层

| 用户层级 | 特征 | 核心诉求 | 产品策略 |
|---|---|---|---|
| 新手用户 | 会写故事，但不会复杂提示词 | 快速开始，尽量少学习 | 默认模板、默认流程、向导式操作 |
| 普通创作者 | 有一定 AI 使用经验 | 想提高稳定性和效率 | 可审阅节点、精修动作、版本回看 |
| 专业作者 / 工作室 | 重视可复用性和稳定出稿 | 希望沉淀流程和模板 | 模板复用、模型档案、版本追踪 |
| 高阶用户 | 关注模型和流程控制 | 想做更细粒度配置 | 开放节点参数、模板复制、自定义配置 |

### 4.2 适配大多数用户的要求

“适配大多数人”不是一句口号，而是明确的产品要求：

- 新用户第一次进入系统后，应能在较短时间内开始第一个项目。
- 用户不需要先理解工作流引擎或插件机制。
- 默认配置必须能覆盖主流中文小说创作场景。
- 产品核心路径必须在浏览器里完成，不依赖复杂本地环境。
- 模型配置要兼容主流供应商，避免锁死在单一平台。

## 5. 产品目标与非目标

### 5.1 第一阶段目标

- 做出一条完整、可运行、可审核的默认创作流程。
- 让用户能从题材输入一路走到成稿导出。
- 让每个关键节点都有明确状态和审核动作。
- 让系统保留版本历史和操作记录。

### 5.2 第一阶段非目标

- 多人实时协作。
- 公开模板市场和插件市场。
- 任意代码执行和开放式第三方插件托管。
- 复杂权限系统、多租户后台、企业级运营能力。
- 向量知识库、RAG 和长时记忆体系。

## 6. 产品原则

### 6.1 节点式创作原则

创作过程必须被拆解成多个节点，而不是单轮大段生成。每个节点都应有明确输入、输出、状态和可执行动作。

### 6.2 人工审核原则

系统不能默认把自动生成结果直接推进到下一阶段。关键节点需要用户确认、精修或重试。

### 6.3 渐进式复杂度原则

- 新手先跑通默认流程。
- 熟练用户再接触模型、提示词和节点配置。
- 高阶用户才进入模板和高级流程定制。

### 6.4 配置资产化原则

模型参数、提示词模板、节点顺序和审核规则都应能沉淀为可复用配置，而不是一次性对话产物。

## 7. 核心概念

| 概念 | 定义 | 备注 |
|---|---|---|
| 项目 | 一次完整的小说创作任务 | 包含目标、流程、产物和版本 |
| 模板 | 一组默认节点顺序和节点配置 | 用于快速开始 |
| 节点定义 | 某个创作步骤的结构定义 | 如大纲生成、章节审核 |
| 节点运行 | 节点在某个项目中的一次执行 | 需要记录输入和输出快照 |
| 内容产物 | 节点输出内容、审核意见、导出文件 | 需要版本管理 |
| 审核动作 | 用户对节点结果做出的操作 | 通过、精修、重试、回退、跳过 |
| Skill | 一组可复用的生成策略和提示词规范 | 第一阶段不做市场化 |
| Hook | 节点生命周期中的受控扩展点 | 第一阶段仅保留接口位 |

## 8. 核心流程

### 8.1 默认创作主流程

1. 创建项目。
2. 录入题材、受众、长度目标和基本设定。
3. 选择默认模板。
4. 生成故事大纲。
5. 审核大纲，必要时精修或重试。
6. 生成章节规划。
7. 逐章生成正文。
8. 对每章执行内容审核和局部精修。
9. 导出 Markdown 成稿。

### 8.2 节点清单

| 节点 | 类型 | 第一阶段 | 说明 |
|---|---|---|---|
| 题材设定 | 输入节点 | 是 | 输入题材、风格、受众、长度目标 |
| 故事大纲 | 生成节点 | 是 | 输出故事主线、冲突和章节骨架 |
| 章节规划 | 生成节点 | 是 | 输出章节列表和摘要 |
| 章节生成 | 生成节点 | 是 | 输出单章正文 |
| 内容审核 | 审核节点 | 是 | 输出问题清单和修改建议 |
| 精修调整 | 编辑节点 | 是 | 基于已有内容局部修改 |
| 人物设计 | 生成节点 | 否 | 作为第二阶段增强能力 |
| 世界观设计 | 生成节点 | 否 | 作为第二阶段增强能力 |

### 8.3 审核动作

| 动作 | 说明 | 结果 |
|---|---|---|
| 通过 | 当前结果可接受 | 推进到下一节点 |
| 精修 | 基于当前结果追加修改指令 | 产生新版本，停留当前节点 |
| 重试 | 放弃当前生成结果重新执行 | 新建一次节点运行 |
| 回退 | 回到上游节点重做 | 当前节点关闭，上游节点重新开放 |
| 跳过 | 当前节点可不执行 | 标记跳过并继续 |

### 8.4 节点状态

| 状态 | 含义 |
|---|---|
| `PENDING` | 待开始 |
| `RUNNING` | 执行中 |
| `WAITING_REVIEW` | 已有结果，等待人工操作 |
| `APPROVED` | 已审核通过 |
| `REFINING` | 精修中 |
| `SKIPPED` | 已跳过 |
| `FAILED` | 执行失败 |

## 9. 功能需求

### 9.1 项目管理

| 编号 | 需求 |
|---|---|
| FR-001 | 用户可以创建、查看、继续和删除创作项目 |
| FR-002 | 项目至少包含题材、风格、目标字数、目标受众等基础信息 |
| FR-003 | 项目需要记录当前工作流进度和最近一次编辑时间 |

### 9.2 模板与流程

| 编号 | 需求 |
|---|---|
| FR-004 | 系统必须提供一条默认创作模板 |
| FR-005 | 用户可以基于默认模板开始项目 |
| FR-006 | 第一阶段允许复制模板并做有限配置调整，不要求自由搭建任意流程 |

### 9.3 节点生成与审核

| 编号 | 需求 |
|---|---|
| FR-007 | 系统必须支持大纲、章节规划、章节正文的生成 |
| FR-008 | 生成结果必须进入待审核状态，而不是自动通过 |
| FR-009 | 用户必须可以执行通过、精修、重试、回退、跳过 |
| FR-010 | 精修必须基于已有结果生成新版本，而不是覆盖旧版本 |

### 9.4 版本与记录

| 编号 | 需求 |
|---|---|
| FR-011 | 每次生成、精修和重试都需要记录版本 |
| FR-012 | 用户需要查看当前节点的历史版本 |
| FR-013 | 系统需要记录审核动作和基本操作日志 |

### 9.5 模型配置

| 编号 | 需求 |
|---|---|
| FR-014 | 系统必须允许用户配置模型供应商和模型档案 |
| FR-015 | 模型配置要与项目或模板关联，便于复用 |
| FR-016 | 第一阶段至少支持一个默认模型路径，并为多模型保留扩展接口 |

### 9.6 导出

| 编号 | 需求 |
|---|---|
| FR-017 | 用户必须可以将项目内容导出为 Markdown |
| FR-018 | 导出结果应包含标题、章节结构和正文内容 |
| FR-019 | `DOCX/PDF` 可以作为增强能力追加，但不应阻塞第一阶段上线 |

### 9.7 需求优先级

为避免后续设计和开发跑偏，第一阶段需求按如下优先级执行：

| 优先级 | 含义 | 处理规则 |
|---|---|---|
| `P0` | 不做就无法形成创作闭环 | 第一阶段必须完成 |
| `P1` | 显著提升体验，但不影响最小闭环存在 | 可在第一阶段后半段或第二阶段早期落地 |
| `P2` | 价值明确，但不应阻塞首版 | 放入后续阶段 |

| 编号 | 优先级 | 说明 |
|---|---|---|
| FR-001 | P0 | 没有项目管理就无法组织创作内容 |
| FR-002 | P0 | 没有基础项目信息就无法稳定驱动生成 |
| FR-003 | P1 | 进度追踪重要，但不是闭环前提 |
| FR-004 | P0 | 默认模板是新手开箱即用的关键 |
| FR-005 | P0 | 基于模板启动项目是主路径 |
| FR-006 | P1 | 模板复制提升复用效率，但不是首个闭环前提 |
| FR-007 | P0 | 大纲、章节规划、正文生成是核心产出链路 |
| FR-008 | P0 | 审核停顿是产品核心价值，不可省略 |
| FR-009 | P0 | 审核动作闭环是关键交互 |
| FR-010 | P0 | 版本化精修是系统差异化能力 |
| FR-011 | P0 | 没有版本记录就无法回看和对比 |
| FR-012 | P1 | 版本查看很重要，但可晚于版本记录落地 |
| FR-013 | P1 | 审计日志重要，但展示形式可简化 |
| FR-014 | P0 | 没有模型配置就无法稳定运行生成链路 |
| FR-015 | P1 | 配置复用提升效率，但可以先做基础关联 |
| FR-016 | P0 | 首版必须为多模型扩展预留接口 |
| FR-017 | P0 | 导出是闭环终点 |
| FR-018 | P0 | 导出内容结构必须完整可用 |
| FR-019 | P2 | `DOCX/PDF` 不是首版阻塞项 |

### 9.8 关键用户场景

下面这些场景是第一阶段设计和验收时必须重点覆盖的代表路径：

| 场景编号 | 典型用户 | 场景描述 | 成功标准 |
|---|---|---|---|
| US-001 | 新手用户 | 第一次进入系统，使用默认模板完成从题材设定到 Markdown 导出的全流程 | 不需要理解高级配置也能完成一次创作闭环 |
| US-002 | 普通创作者 | 对某一章结果不满意，执行精修并回看新旧版本差异 | 至少可以保留两个版本并继续推进流程 |
| US-003 | 普通创作者 | 大纲审核不通过，回退后重新生成，再继续章节规划 | 回退不会导致项目状态混乱或内容丢失 |
| US-004 | 工作室用户 | 基于已有模板创建新项目并替换模型档案 | 模板与模型配置可以复用，不需要重复填写全部参数 |
| US-005 | 高阶用户 | 在默认模板基础上调整部分节点提示词和参数 | 能修改配置，但不会破坏默认主路径 |

### 9.9 异常与失败场景

除了正向流程，系统还必须覆盖以下失败路径：

| 场景编号 | 异常场景 | 系统要求 |
|---|---|---|
| EX-001 | 模型调用超时 | 节点标记为失败，可重试，错误原因可见 |
| EX-002 | 模型返回内容质量差 | 用户可直接发起精修或重试，不需要离开当前节点 |
| EX-003 | 用户误操作回退 | 系统需保留当前版本快照，避免内容直接丢失 |
| EX-004 | 页面刷新或浏览器关闭 | 重新进入后可恢复到最近一次持久化状态 |
| EX-005 | 导出失败 | 不影响项目正文内容，用户可重新触发导出 |
| EX-006 | 模型配置无效或密钥错误 | 在运行前给出明确错误，避免节点进入假成功状态 |
| EX-007 | 跳过节点后继续流程 | 系统需保留跳过记录，后续可追溯 |

## 10. 用户体验需求

### 10.1 新手路径

新手用户的核心目标不是“学会系统全部能力”，而是“尽快完成第一篇可用内容”。因此，新手路径必须满足：

- 进入系统后能立即看到“新建项目”。
- 默认模板可以直接使用。
- 关键节点提供清晰动作按钮，而不是隐藏在复杂菜单里。
- 不要求用户理解提示词工程术语。

### 10.2 进阶路径

普通创作者和进阶用户需要：

- 调整模型和参数。
- 修改节点提示词。
- 复制模板并做个人化修改。
- 追踪同一章节的不同版本。

### 10.3 高阶路径

高阶用户可以逐步接触：

- 自定义模板。
- 更细粒度的节点参数。
- 更复杂的审核规则。

但这些能力不应干扰新手默认路径。

- 左侧为节点列表和节点状态。

- 审核动作必须始终可见，不能埋得太深。

## 13. 非功能需求

### 13.1 可恢复性

- 节点运行到 `WAITING_REVIEW` 后必须可持久化恢复。
- 用户刷新页面后不能丢失当前节点状态和最近产物。

### 13.2 可追溯性

- 每次生成都要记录输入快照、模型快照、时间、耗时。
- 每次精修和重试都必须生成新版本。

### 13.3 可观测性

- 需要记录节点级别成功率、失败原因、耗时、token 用量和成本。
- 任何模型调用失败都应能定位到具体项目和节点运行。

### 13.4 安全性

- 模型密钥不能明文写入代码或日志。
- 项目数据和导出文件要与用户项目绑定，避免串读。

### 13.5 性能与体验

- 用户触发节点后应立刻看到“执行中”状态。
- 长文本生成要支持流式返回、轮询或其他可感知进度的方式。

## 14. 第一阶段范围与后续阶段

### 14.1 第一阶段必须交付

- 默认模板
- 节点生成闭环
- 审核动作闭环

### 14.2 第二阶段候选

## 15. 验收口径

| 里程碑 | 范围 | 验收标准 |
|---|---|---|
| M1 项目骨架 | 项目、模板、基础页面 | 能创建项目并进入默认工作流 |
| M2 生成闭环 | 大纲、章节、审核动作 | 至少一条默认流程可以完整跑通 |
| M3 版本与导出 | 版本追踪、Markdown 导出 | 能查看历史版本并导出成稿 |
| M4 模型配置 | 提供商配置与模型档案 | 能切换模型档案并完成一次创作 |

## 16. 当前结论

- easyStory 的本质是“节点式 AI 小说创作系统”，不是普通聊天机器人。
- 它的第一性目标是“让更多普通用户也能稳定写作”，不是优先满足少数高级玩家的平台需求。
- 需求核心不是“多做功能”，而是“把默认创作闭环做稳、做清楚、做可审阅”。
