## 1. 结论

从新项目视角，不参考当前前端页面实现，easyStory 最合适拆成 **12 个一级功能页面**，另有 **1 个认证页**。

计数口径：

- “一级功能页面”指有独立路由、独立主工作区布局和独立任务心智的页面。
- 页内页签、右侧抽屉、详情弹窗、向导步骤不单独计入页面数量。
- “系统配置台”只对管理员或高级维护者显示；普通用户日常主路径实际是 11 个功能页面。
- 登录 / 注册是产品入口，不计入创作业务页。

推荐页面如下：

| 序号 | 页面 | 建议路由 | 主职责 | 主要后端依据 |
|---:|---|---|---|---|
| 0 | 认证页 | `/auth` | 登录、注册、进入工作区 | `user` |
| 1 | 项目入口页 | `/workspace` | 项目列表、状态扫描、回收站、进入项目 | `project` |
| 2 | 新建孵化页 | `/workspace/new` | 模板选择、自由文本 / 对话生成设定草稿、创建项目 | `project incubator`、`template` |
| 3 | 项目总览页 | `/projects/:projectId` | 项目健康度、下一步动作、进度、成本和异常总览 | `project`、`workflow`、`content`、`billing` |
| 4 | 创作准备页 | `/projects/:projectId/foundation` | `ProjectSetting`、大纲、开篇设计、章节任务 | `project`、`content`、`workflow` |
| 5 | 文稿工作台页 | `/projects/:projectId/studio` | 项目文稿树、正文编辑、AI 对话、版本管理 | `project documents`、`content`、`assistant` |
| 6 | 工作流控制页 | `/projects/:projectId/workflows` | 启动 / 暂停 / 恢复 / 取消、节点监控、日志、Prompt 回放 | `workflow`、`observability`、`review`、`billing` |
| 7 | 质量与连续性页 | `/projects/:projectId/quality` | 审核问题、Story Bible、上下文报告、stale 处理 | `review`、`context`、`content` |
| 8 | 分析实验室页 | `/projects/:projectId/analysis` | 小说导入 / 采样 / 分析 / Skill 生成和试写 | `analysis`、`assistant skills` |
| 9 | 导出页 | `/projects/:projectId/exports` | 导出预检、范围 / 格式 / 版本策略、导出记录下载 | `export`、`content`、`workflow` |
| 10 | 项目设置页 | `/projects/:projectId/settings` | 项目元信息、项目规则、项目 AI 偏好、项目 Skill / MCP、项目凭证、安全操作 | `project`、`assistant`、`credential`、`observability` |
| 11 | 个人设置页 | `/settings` | 个人规则、AI 偏好、模型连接、个人 Skill / Agent / Hook / MCP、费用概览 | `assistant`、`credential`、`billing`、`user` |
| 12 | 系统配置台 | `/admin/config` | 系统内置 Skills / Agents / Hooks / MCP / Workflows / Templates 管理 | `config_registry`、`template` |

## 2. 设计原则

### 2.1 按用户工作阶段拆，不按后端模块拆

后端模块是维护边界，页面是用户任务边界。页面不应该变成 `project / content / workflow / assistant` 一模块一页，否则用户会在“写一章”时来回跳转。

更合理的任务阶段是：

```text
进入项目 -> 孵化设定 -> 确认前置资产 -> 写正文 -> 跑工作流 -> 修质量 -> 导出
```

### 2.2 主路径优先，增强能力后置

系统定位是 AI 小说创作平台，不是通用 Agent 控制台。页面层级应把“设定、大纲、章节、审查、修订、连续性”放在主导航，把 Skill / Agent / Hook / MCP 放进设置和高级配置。

### 2.3 同一条真值只能有一个主编辑入口

同一个数据可以在多个页面出现，但只能有一个主编辑心智：

| 数据 | 主编辑页 | 其他页面展示方式 |
|---|---|---|
| `ProjectSetting` | 创作准备页 / 项目设置页中的摘要入口 | 总览页只展示健康度和缺口 |
| `Outline` / `OpeningPlan` | 创作准备页 | 文稿工作台可作为 canonical 文稿读取 |
| 正文章节 | 文稿工作台页 | 工作流控制页展示节点产物和状态 |
| 项目规则 / AI 偏好 | 项目设置页、个人设置页 | 文稿工作台只显示当前生效连接和模式 |
| 系统内置配置 | 系统配置台 | 新建孵化页和工作流控制页只消费配置结果 |

### 2.4 复杂页面靠布局分区，不靠无限拆页

文稿工作台、工作流控制、质量连续性都很重，但它们的用户任务不同。应该拆成不同一级页面；每页内部再用稳定分区承载复杂度。

## 3. 页面详设

### 3.1 认证页

职责：

- 登录、注册、保存 token 后进入工作区。
- 处理 token 失效后的重新登录。
- 不承载项目、凭证、模型配置等业务内容。

布局建议：

| 区域 | 内容 |
|---|---|
| 左侧 / 顶部品牌区 | 产品名、简短定位、当前版本或部署环境提示 |
| 表单区 | 登录 / 注册切换、用户名、密码、提交、错误提示 |
| 底部辅助区 | 本地部署提示、隐私提示 |

为什么单独存在：

- 认证是进入系统的边界，不应和项目列表混排。
- 后端 `user` 只提供注册 / 登录，登出由前端清 token，页面逻辑很薄。

### 3.2 项目入口页

职责：

- 展示当前用户所有项目。
- 支持搜索、题材 / 状态筛选、最近打开、按更新时间排序。
- 展示每个项目的核心状态：项目名、题材、目标字数、当前章节数、最近工作流状态、是否有 stale / paused / failed。
- 提供新建项目入口、恢复上次项目入口。
- 承载回收站：软删除项目列表、恢复、彻底删除、清空回收站。

布局建议：

| 区域 | 内容 |
|---|---|
| 顶部操作栏 | 搜索、筛选、新建项目、个人设置入口 |
| 主列表区 | 项目卡片或表格；每项显示进度、状态、最近动作 |
| 右侧状态栏 | 最近项目、失败 / 暂停提醒、系统连接状态 |
| 回收站抽屉 | deleted-only 项目、恢复、物理删除、清空 |

为什么这样设计：

- `Project` 是用户所有创作资产的入口，项目列表需要快速扫描和恢复。
- 回收站属于项目集合管理，不值得拆成独立一级页。
- 项目入口不应放具体写作工具，否则用户还没选项目就进入过深功能。

### 3.3 新建孵化页

职责：

- 支持三种建项方式：模板快速开始、自由文本设定、对话式设定草稿。
- 调用孵化接口生成结构化 `ProjectSetting` 草稿。
- 展示完整度、缺失字段、追问建议、未映射回答。
- 让用户确认项目名、题材、目标篇幅、模板 / workflow。
- 创建项目并生成一次性默认项目文稿模板。

布局建议：

| 区域 | 内容 |
|---|---|
| 左侧步骤栏 | 方式选择、输入设定、草稿确认、创建项目 |
| 中央输入区 | 模板问题、自由文本输入、对话记录或聚合输入 |
| 右侧结构化预览 | `ProjectSetting` 草稿、完整度、追问、风险提示 |
| 底部确认区 | 创建项目、保存草稿、返回项目入口 |

为什么单独存在：

- 孵化阶段是“项目尚不存在或尚未确认”的状态，不能塞进项目内设置页。
- 模板、自有文本、对话提取最终都产出同一个 `ProjectSetting`，放在同一页能避免三套创建入口。
- 新建页需要强引导；进入项目后则应变成工作台，不应继续停留在表单向导心智。

### 3.4 项目总览页

职责：

- 作为项目内首页，回答“现在这个项目该做什么”。
- 展示创作准备状态：设定完整度、大纲状态、开篇设计状态、章节任务状态。
- 展示当前或最近工作流：created / running / paused / failed / completed。
- 展示正文进度：章节数、已确认章节、stale 章节、字数。
- 展示质量和风险：审核失败数、Story Bible 冲突、导出阻断项。
- 展示成本摘要：今日 token、最近工作流消耗、预算告警。
- 提供下一步动作：补设定、生成大纲、打开正文、恢复工作流、处理失败、导出。

布局建议：

| 区域 | 内容 |
|---|---|
| 顶部项目头 | 项目名、题材、状态、目标字数、关键操作 |
| 左侧主状态 | 准备链路进度、正文进度、当前工作流 |
| 中央下一步 | 系统推荐动作、阻断原因、快捷入口 |
| 右侧健康栏 | 成本、质量、stale、最近错误 |
| 底部动态 | 最近编辑、最近生成、最近导出 |

为什么需要：

- 如果没有项目总览，用户每次进入项目都必须判断该去准备页、正文页还是工作流页。
- 总览页只做状态汇聚和导航，不直接承载长表单或编辑器，避免成为杂物页。

### 3.5 创作准备页

职责：

- 建立和维护标准前置资产链路：

```text
ProjectSetting -> Outline -> OpeningPlan -> ChapterTask
```

- 编辑 / 保存 `ProjectSetting`，展示完整度和影响评估。
- 生成、编辑、批准大纲。
- 生成、编辑、批准开篇设计。
- 拆分、查看、编辑、重建章节任务。
- 对上游资产变更后的 stale / impact 做集中处理。

布局建议：

| 区域 | 内容 |
|---|---|
| 左侧资产导航 | 项目设定、大纲、开篇设计、章节任务 |
| 中央编辑区 | 当前资产编辑器；结构化表单 + Markdown / 文本编辑按资产切换 |
| 右侧检查区 | 完整度、依赖状态、影响范围、版本历史 |
| 底部动作条 | 生成、保存草稿、批准、重建章节任务、查看版本 |

页面内分区建议：

| 分区 | 具体功能 |
|---|---|
| 项目设定 | 固定 schema 字段、自由补充说明、从项目说明提取、完整度检查 |
| 大纲 | 获取、生成、保存草稿、批准、版本列表 |
| 开篇设计 | 获取、生成、保存草稿、批准、版本列表 |
| 章节任务 | 章节任务表格、章节标题 / brief / key characters / key events、重建任务、stale 状态 |
| 影响处理 | 上游变更后哪些资产需要 stale、哪些可继续使用、哪些需用户确认 |

为什么大纲、开篇设计、章节任务不拆成三页：

- 它们是同一条前置资产链，不是三个互不相关的内容库。
- 页面拆太细会导致用户难以判断“为什么章节任务需要重建”。
- 统一在创作准备页，右侧依赖状态和影响面板可以复用。

### 3.6 文稿工作台页

职责：

- 高频写作与编辑主界面。
- 展示统一项目文稿树：设定、数据层、章节规划、时间轴、附录、导出工作稿，以及 canonical 大纲 / 开篇 / 正文章节入口。
- 支持 Markdown 文稿编辑、JSON 文稿编辑和 JSON 预览。
- 支持章节正文读取、保存草稿、批准、版本列表、回滚、标记最佳版本。
- 支持项目文稿文件创建、重命名、删除。
- 承载项目内 AI 聊天：当前会话、当前文稿上下文、可选 Skill / Agent / Hook / MCP、工具读写授权。
- 展示编辑态保存状态、版本、字数、当前模型连接。

布局建议：

| 区域 | 内容 |
|---|---|
| 左侧文稿树 | 目录、搜索、新建文件 / 目录、重命名、删除、canonical 只读标识 |
| 中央编辑器 | Markdown / JSON / 章节正文编辑器、预览切换、保存状态、字数 |
| 右侧 AI 助手 | 会话、模型连接、Skill 模式、选中文稿上下文、工具调用结果 |
| 右侧次级抽屉 | 版本历史、diff、回滚、最佳版本、审核问题 |
| 底部状态条 | 当前 path、source、version、dirty 状态、最近保存时间 |

页面内必须区分的对象：

| 对象 | 行为 |
|---|---|
| 文件层文稿 | 可编辑、可重命名、可删除、可由 assistant 授权写回 |
| canonical 大纲 / 开篇 | 可读取；主编辑入口仍是创作准备页 |
| canonical 正文章节 | 可编辑；正文真值走 `contents + content_versions` |
| 目录占位 | 只用于树结构，不承载正文内容 |
| 数据层 JSON | 可编辑 JSON；关系图只做预览，不替代 JSON 真值 |

为什么文稿工作台必须单独成页：

- 它需要最大主视图给编辑器，和工作流监控的时间线 / 日志布局冲突。
- assistant 聊天在这里是写作上下文的一部分，不应变成全局独立聊天页。
- 项目文稿工具要求模型不关心底层 DB / 文件层，统一文稿树最适合作为用户可见心智。

### 3.7 工作流控制页

职责：

- 管理项目工作流执行：启动、暂停、恢复、取消。
- 列出历史工作流执行和当前 active 执行。
- 展示节点列表 / 时间线：pending、running、reviewing、fixing、interrupted、completed、failed、skipped。
- 展示章节任务执行进度。
- 通过 SSE 显示实时事件。
- 展示节点详情：输入、输出、上下文报告、审核结果、精修结果、错误、token 消耗。
- 提供 Prompt / 响应回放、执行日志、审计相关入口。
- 处理暂停原因：用户中断、预算超限、审核失败、错误、loop pause、章节上限。

布局建议：

| 区域 | 内容 |
|---|---|
| 顶部控制区 | 工作流选择、启动、暂停、恢复、取消、模式 / 预算摘要 |
| 左侧执行列表 | 当前执行、历史执行、状态筛选 |
| 中央时间线 | 节点和章节循环进度、实时状态、耗时、token |
| 右侧详情面板 | 节点输入 / 输出、上下文、审核、日志、Prompt 回放 |
| 底部事件流 | SSE 事件、错误、重试、恢复提示 |

为什么不和文稿工作台合并：

- 工作流页关注“执行控制和可观测性”，文稿工作台关注“编辑和对话”。
- 节点详情、Prompt 回放、日志、预算都需要可展开的诊断面板，会压缩编辑器空间。
- 手动模式下生成结果可以跳转到文稿工作台确认，自动模式下用户主要留在控制页观察。

### 3.8 质量与连续性页

职责：

- 跨章节集中处理质量、连续性和上下文风险。
- 汇总 ReviewAction：按章节、严重级别、reviewer、状态筛选。
- 展示审核问题列表：定位、证据、建议修复、处理状态。
- 管理 Story Bible 事实：人物状态、地点、时间线、设定变更、伏笔、关系。
- 处理事实冲突：potential / confirmed、替代链、是否注入上下文。
- 展示 stale 传播：哪些章节基于旧设定 / 旧大纲 / 旧事实。
- 查看上下文构建报告：section、token、裁剪、缺失、overflow。
- 提供批量动作：重新审核、跳转章节、标记已处理、重新生成范围。

布局建议：

| 区域 | 内容 |
|---|---|
| 顶部筛选区 | 章节范围、问题类型、严重级别、状态、reviewer |
| 左侧问题队列 | 审核 issue、stale 章节、事实冲突 |
| 中央详情区 | 问题正文、证据、建议、相关章节片段 |
| 右侧连续性面板 | Story Bible、事实链、上下文注入情况 |
| 底部批量操作 | 重审、跳转编辑、忽略、标记、重建章节任务 |

为什么需要独立页：

- 审核和 Story Bible 都是跨章节问题，不适合只藏在单章编辑器侧栏。
- 工作流控制页只能看到某次执行中的质量结果；质量与连续性页要看到项目全局。
- 该页让“长期一致性”成为产品主能力，而不是日志里的附属信息。

### 3.9 分析实验室页

职责：

- 上传或导入已有小说文本。
- 章节切分、编码检测、采样策略选择。
- 发起分析：plot、character、style、pacing、structure。
- 展示分析结果、分析范围、来源标题、建议。
- 编辑分析报告中的用户备注。
- 基于确认后的分析生成 Skill。
- 试写、调整、保存 Skill 到个人或项目作用域。
- 管理分析记录：列表、最新结果、详情、删除。

布局建议：

| 区域 | 内容 |
|---|---|
| 顶部任务栏 | 上传 / 导入、分析类型、作用域、创建分析 |
| 左侧来源区 | 文本来源、章节切分、采样范围 |
| 中央报告区 | 结构化分析结果、建议、用户备注 |
| 右侧 Skill 区 | 生成 Skill、试写内容、保存作用域、关联记录 |
| 底部历史区 | 分析记录列表、筛选、删除 |

为什么单独存在：

- 分析是“从外部文本提炼能力”，不是当前项目正文编辑。
- 它最终产出可被主流程消费的 Skill 或风格参考，但分析过程本身需要上传、采样和确认，不应挤进设置页。

### 3.10 导出页

职责：

- 导出前预检章节状态。
- 选择导出范围：整本书、部分章节、单章、设定资料。
- 选择格式：TXT、Markdown、DOCX 等。
- 选择版本策略：latest、specific、best。
- 处理异常章节：draft、failed、skipped、generating、stale。
- 配置导出模板和元信息。
- 展示导出记录、文件大小、创建时间、下载。

布局建议：

| 区域 | 内容 |
|---|---|
| 左侧范围选择 | 章节树、状态、勾选范围 |
| 中央预检报告 | 阻断项、warning、可导出内容、处理选项 |
| 右侧导出配置 | 格式、模板、版本策略、元信息 |
| 底部记录区 | 历史导出、下载、重新导出 |

为什么单独存在：

- 导出需要完整预检和范围选择，和正文编辑不是同一操作节奏。
- 导出记录是项目资产，应该可回查和下载。
- `generating` 阻断、`stale` warning、`best` 缺失等状态需要集中解释。

### 3.11 项目设置页

职责：

- 编辑项目元信息：名称、题材、目标字数、状态、系统凭证池开关。
- 管理项目长期规则 `AGENTS.md`。
- 管理项目 AI 偏好 `preferences.yaml`。
- 管理项目 Skills。
- 管理项目 MCP。
- 管理项目级模型凭证和验证状态。
- 查看项目审计日志。
- 安全操作：归档 / 删除 / 恢复相关入口、危险操作确认。

布局建议：

| 区域 | 内容 |
|---|---|
| 左侧设置导航 | 基础信息、规则、AI 偏好、Skills、MCP、凭证、审计、危险操作 |
| 中央表单区 | 当前设置项主编辑区 |
| 右侧说明区 | 当前作用域、继承 / 覆盖关系、验证状态、风险提示 |
| 底部保存条 | 未保存提示、保存、重置 |

为什么项目设置和个人设置要拆开：

- 项目配置是“这个项目如何覆盖个人默认”，个人配置是“我所有项目的默认习惯”。
- 项目设置需要展示继承关系和项目影响范围；个人设置需要展示全局连接和个人能力库。
- 合并会导致作用域切换复杂，用户容易误改全局配置。

### 3.12 个人设置页

职责：

- 个人资料和认证状态。
- 管理个人长期规则。
- 管理个人 AI 偏好。
- 管理个人模型连接：provider、api_dialect、base_url、default_model、验证、启用 / 禁用、删除。
- 展示工具能力验证：基础连接、流式工具、非流工具。
- 管理个人 Skills / Agents / Hooks / MCP。
- 查看个人 token 使用和费用概览。
- 退出登录。

布局建议：

| 区域 | 内容 |
|---|---|
| 左侧设置导航 | 账户、AI 偏好、模型连接、规则、Skills、Agents、Hooks、MCP、费用 |
| 中央列表 / 表单 | 当前资源列表、详情编辑、验证操作 |
| 右侧状态区 | 连接验证状态、作用域说明、当前默认模型 |
| 底部操作区 | 保存、删除、重新验证、退出 |

为什么个人增强能力放这里：

- Assistant 主路径默认只需要规则、当前会话、文稿上下文和模型连接。
- Skill / Agent / Hook / MCP 是显式增强层，适合放在设置里维护，再在文稿工作台按需选择。
- 凭证涉及安全、验证和审计，不应该散落在各个创作页面。

### 3.13 系统配置台

职责：

- 管理系统内置配置资源：Skills、Agents、Hooks、MCP Servers、Workflows。
- 管理模板：内置模板、可创建模板、模板节点、引导问题。
- 支持列表、详情、JSON / YAML 编辑、保存前 staged 校验。
- 暴露 strict DTO 校验失败、配置加载失败、未知字段错误。
- 只对管理员或具备配置写权限的用户开放写操作。

布局建议：

| 区域 | 内容 |
|---|---|
| 左侧资源导航 | Skills、Agents、Hooks、MCP、Workflows、Templates |
| 中央列表区 | 资源列表、搜索、分类、状态 |
| 右侧详情区 | 表单视图、源码视图、校验结果 |
| 底部校验区 | staged full-config 校验、错误定位、保存 |

为什么不让普通用户直接在主路径看到：

- 系统配置是平台能力维护，不是日常写作心智。
- 普通用户自己的 Skills / Agents / Hooks / MCP 已由个人设置承载。
- 系统配置台如果进入主导航，会把产品误导成通用 agent 管理器。

## 5. 不建议的拆分和合并

### 5.1 不建议把创作准备、文稿工作台、工作流控制合成一页

这三类页面都很重，但主视图诉求完全不同：

| 页面 | 主视图 |
|---|---|
| 创作准备 | 前置资产编辑和依赖状态 |
| 文稿工作台 | 大编辑器和 AI 对话 |
| 工作流控制 | 执行时间线、节点详情、日志 |

合并会导致布局互相抢空间，也会让用户分不清当前是在“准备资产”、“写正文”还是“监控执行”。

### 5.2 不建议把大纲、开篇设计、章节任务拆成三个一级页

这三者是同一条依赖链。拆成三个页面会削弱“上游变更导致下游 stale”的可见性。它们更适合在创作准备页中用左侧资产导航和右侧影响面板组织。

### 5.3 不建议把 Assistant 做成独立主页面

普通聊天必须存在，但它应该绑定当前项目、当前文稿、当前上下文。独立全局聊天会让用户误以为 easyStory 是通用聊天工具，也会削弱项目文稿工具的稳定边界。

### 5.4 不建议把日志、Prompt 回放、审计都拆成独立页

执行日志和 Prompt 回放应该绑定工作流 / 节点详情；审计日志应该绑定设置里的安全或管理操作。单独拆日志页会让问题定位脱离上下文。

### 5.5 不建议把模型凭证散落到每个生成按钮旁边

生成按钮附近可以显示当前连接和验证状态，但创建、修改、验证、启用 / 禁用凭证必须回到设置页。凭证涉及安全、作用域和审计，散落在业务页会造成误操作。

### 5.6 不建议把模板库单独做普通用户一级页

模板主要在新建孵化时消费，在系统配置台维护。普通创作用户不应频繁进入模板管理页；否则会把“写小说”变成“调系统配置”。

## 6. 推荐导航结构

全局层：

```text
认证页
  -> 项目入口页
  -> 新建孵化页
  -> 个人设置页
  -> 系统配置台（管理员）
```

项目层：

```text
项目总览
创作准备
文稿工作台
工作流控制
质量与连续性
分析实验室
导出
项目设置
```

项目层导航建议保持固定顺序：

```text
总览 -> 准备 -> 文稿 -> 工作流 -> 质量 -> 分析 -> 导出 -> 设置
```

理由：

- 顺序贴合创作生命周期。
- 高频页面“文稿工作台”在中间，前后分别是准备和工作流。
- 设置放最后，避免主路径被配置项打断。

## 7. 最小可落地版本

如果需要分阶段实现，建议不要按“页面数量先减半”处理，而是按每页能力收缩。

第一阶段仍保留页面骨架：

| 页面 | 第一阶段保留内容 | 可后置内容 |
|---|---|---|
| 项目入口页 | 项目列表、新建入口、回收站 | 高级筛选 |
| 新建孵化页 | 模板 / 自由文本生成设定草稿 | 完整多轮对话体验 |
| 项目总览页 | 准备状态、下一步、active workflow | 复杂趋势图 |
| 创作准备页 | `ProjectSetting`、大纲、开篇、章节任务 | 批量影响处理 |
| 文稿工作台页 | 文稿树、编辑器、AI 聊天、版本 | 高级 diff |
| 工作流控制页 | 启动 / 暂停 / 恢复、节点列表、事件 | 图形化编排 |
| 质量与连续性页 | 审核问题、Story Bible 列表 | 复杂关系可视化 |
| 分析实验室页 | 分析记录、生成 Skill | 上传源文件长期保留 |
| 导出页 | TXT / Markdown 导出预检 | DOCX / EPUB / 高级排版 |
| 项目设置页 | 基础信息、规则、AI 偏好、凭证 | 项目级高级 MCP |
| 个人设置页 | 凭证、偏好、规则、Skills | 复杂 Agent / Hook 编辑器 |
| 系统配置台 | 只读列表 + 管理员编辑 | 可视化 workflow 设计器 |

这样做的好处是导航结构从第一天就稳定，后续只是在页面内补能力，不需要反复迁移用户心智。

## 8. 最终判断

easyStory 的最佳页面数不是越少越好。少于 8 个一级功能页时，创作准备、正文编辑、工作流监控、质量连续性会被迫混在一起，核心页面会变成不可维护的大面板。超过 15 个页面时，又会把同一创作链路拆碎，用户需要频繁跳转。

因此推荐：

- **12 个一级功能页面**
- **1 个认证页**
- **项目内 8 个页面**
- **全局 / 管理 4 个页面**

这个划分能覆盖当前后端能力，又不会让普通写作主路径被配置系统和运行时细节淹没。
