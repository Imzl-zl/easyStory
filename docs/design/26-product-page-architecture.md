## 1. 结论

easyStory 需要 **5 个主工作空间**。

> **术语说明**：这里用"主工作空间"而非"页面"，因为一个工作空间可能对应多条工程路由（如书架工作空间包含 `/workspace/lobby`、`/workspace/lobby/new`、`/workspace/lobby/settings` 等），但它们共享同一套导航心智和布局骨架。产品上用户感知到的是 5 个工作空间，工程上路由数量可以大于 5。详见 [§7 路由过渡口径](#7-路由过渡口径)。

| 序号 | 工作空间 | 核心路由 | 存在理由 |
|---:|---|---|---|
| 1 | 认证 | `/auth` | 系统边界，登录/注册 |
| 2 | 书架 | `/workspace` | 项目入口、创建、回收站、模板库、全局设置 |
| 3 | 创作工作室 | `/workspace/project/[id]/studio` | **主场**。80% 时间在这里。写作 + AI 对话 + 全部创作能力 |
| 4 | 工作流引擎 | `/workspace/project/[id]/engine` | 自动化批量执行和监控 |
| 5 | 分析实验室 | `/workspace/project/[id]/lab` | 从外部文本提炼风格、结构、Skill |

不在主工作空间列表里的东西：

- **新建孵化** — 不是独立工作空间，是书架里的一次对话。说一句"我想写个XX故事"，AI 聊两轮，项目就建好了。
- **创作准备（设定/大纲/开篇）** — 不是独立工作空间，是工作室文档树里的文件。边写边聊边生成，不是先填表再写。
- **质量与连续性** — 不是独立工作空间，是工作室内的四层可见性体系：文档树标记 + 编辑器内标注 + 审校面板汇总 + AI 聊天解释。详见 [§3.3.4](#334-质量连续性四层可见性)。
- **导出** — 不是独立工作空间，是工作室里的一个操作。选中范围、选格式、导出下载。
- **项目总览** — 不需要。进入项目直接落地工作室，上次写到哪光标就在哪。
- **系统配置台** — 管理员入口，不在普通用户导航里。

## 2. 设计哲学

### 2.1 创作是对话，不是流水线

一个写作者的真实路径：

```
打开工具 → 有模糊想法 → 跟 AI 聊 → 聊着聊着清晰了 → 开始写
→ 写着写着又想聊 → 聊完继续写 → 写到一段落想看看结构 → 调出大纲面板瞄一眼
→ 觉得可以让 AI 批量生成后续章节 → 去引擎跑一下 → 回来继续写
```

不是：

```
孵化页填设定 → 准备页做大纲 → 工作台写正文 → 工作流页跑审批 → 质量页修问题 → 导出页导出
```

前者是创作。后者是工厂。

### 2.2 工作空间 ≠ 工作阶段

一个工作空间是一个完整的活动场域，不是流水线上的一道工序。

- **书架**：管理项目的地方。你可以在这创建、搜索、删除、配置。产品心智上是一个空间，工程上可能对应多条路由（`/workspace/lobby`、`/workspace/lobby/settings` 等）。
- **工作室**：写作的地方。所有跟"写"有关的能力都在这里——编辑、聊天、大纲、设定、版本、导出。一条核心路由 + 面板/对话框承载子能力。
- **引擎**：自动化执行的地方。批量生成、自动审核、格式导出——后台跑，不打断写作。
- **实验室**：研究外部文本的地方。上传小说、分析风格、提炼 Skill——做完就回工作室用。

术语上本文档用"工作空间"描述产品心智，用"路由/面板/对话框"描述工程实现。一个工作空间在工程上可能拆分为多条路由（支持刷新、深链、未保存保护），但这不影响用户感知到的空间数量。

### 2.3 能力按需调出，不按阶段切页

| 你想做的事 | 怎么做（新设计） | 旧设计的做法 |
|---|---|---|
| 想写新项目 | 书架页跟 AI 聊一句，建好直接进工作室 | 跳转到孵化页填表单 |
| 看看大纲 | 工作室左侧文档树点一下 `大纲/总大纲.md` | 跳转到准备页 |
| 写正文 | 工作室中间编辑器直接写 | 跳转到文稿工作台 |
| 批量生成章节 | 工作室顶部点"引擎"，选工作流跑起来，切回来继续写 | 跳转到工作流控制页守着 |
| 导出 | 工作室文档树右键 → 导出，或者工具栏 → 导出 | 跳转到导出页 |
| 检查一致性 | 聊天框里 @AI "帮我检查前后矛盾"，有结果在聊天里看 | 跳转到质量页看报告 |
| 分析参考小说 | 去实验室上传分析，产出 Skill 回到工作室用 | 一样但多了很多中间页 |
| 改模型配置 | 书架页右上角 → 设置 | 跳转到个人设置页 |

### 2.4 真值源唯一，但编辑入口自然

| 数据 | 在哪编辑 | 为什么 |
|---|---|---|
| 章节正文 | 工作室编辑器 | 大部分时间在写正文 |
| 大纲 | 工作室文档树 `大纲/总大纲.md` | 大纲也是文档，在文档树里打开编辑 |
| 开篇设计 | 工作室文档树 `大纲/开篇设计.md` | 同上 |
| 项目设定（结构化字段） | 工作室 `项目说明.md` 自动提取 + AI 聊天中自然描述 | 你不是在填表，你是在描述故事 |
| 数据层 JSON | 工作室 JSON 编辑器 | 结构化数据有专门编辑体验 |
| 项目规则/AI偏好 | 书架页 → 设置面板 | 低频，不是每次创作都要改 |
| 模型凭证 | 书架页 → 设置面板 | 安全相关，不应散落 |
| 系统配置 | 管理后台 | 普通用户不可见 |

## 3. 页面详设

### 3.1 认证页 `/auth`

极简。登录/注册切换，登完直接进书架。

```

│           easyStory                      │
│         AI 小说创作平台                   │
│                                          │
│   [用户名] [密码] [登录 / 注册]           │
│                                          │
│   本地部署 · 数据由你掌控                 │
└──────────────────────────────────────────┘
```

---

### 3.2 书架页 `/workspace`

**定位**：项目司令部。管理所有项目、创建新项目、个人设置。

**布局**（参考现有 Lobby 设计）：

```
┌──────────────────────────────────────────────────────────┐
│ [easyStory]  [书架] [←当前项目] [设置]  [头像] [退出]    │
├────────────┬─────────────────────────────────────────────┤
│ 左侧导航   │  项目卡片网格                                │
│            │                                             │
│ 我的作品   │  ┌──────┐ ┌──────┐ ┌──────┐               │
│ 模板库     │  │ 项目1 │ │ 项目2 │ │ 项目3 │               │
│ 回收站     │  │ 修仙  │ │ 悬疑  │ │ 科幻  │               │
│ 设置       │  │ 12章  │ │ 5章   │ │ 新建  │               │
│            │  └──────┘ └──────┘ └──────┘               │
│            │                                             │
│            │  [搜索] [筛选] [新建项目]                    │
└────────────┴─────────────────────────────────────────────┘
```

**新建项目的方式**（不跳页）：

1. **快速创建**：点击"新建项目"，弹出对话框。输入一句话描述或选模板，AI 在后台孵化，1-2 轮对话后项目建好，自动进入工作室。整个过程不需要离开书架页。
2. **从模板创建**：模板库里选一个，一键创建。
3. **导入创建**：上传已有文稿，系统自动解析创建项目。

**每个项目卡片显示**：
- 项目名、题材、状态（草稿/连载中/已完成）
- 章数、字数、上次编辑时间
- 工作流状态（如有 active execution 亮起小绿点）
- 一键"继续创作"进入工作室

**为什么孵化不是独立页**：

孵化是"项目创建过程"，时长 1-3 轮对话，不值得一个全屏页面。它是一个创建流程中的过渡态——书架页里弹出对话面板，聊完项目就建好了，用户直接落地工作室。用户甚至感觉不到经过了"创建流程"——他只是在聊天，聊着聊着项目就有了。

### 3.3 创作工作室 `/workspace/project/[id]/studio`

**定位**：唯一主场。写作者 80% 时间在这里。

**设计参考**：现有 Studio 页面已验证了核心布局的合理性。

**工程可行性**：本文描述的"大纲/开篇/章节像文档一样在树里打开编辑"不是空想。当前代码已支持这种模型——`大纲/总大纲.md`、`大纲/开篇设计.md`、`正文/第xxx章.md` 会被 `studio-document-support.ts` 解析为 DB 真值目标并通过 content API 读写，普通项目文稿走文件层保存。工作室文档树是这两种存储后端的统一视图。

**布局**：

```
┌──────────────────────────────────────────────────────────────┐
│ [←书架] [项目名]  [正文/设定/大纲]面包屑   [保存] [聊天] [引擎] │
├──────────┬────────────────────────┬─────────────────────────┤
│ 文档树   │  编辑器                 │  AI 共创助手            │
│ (236px)  │  (自适应)              │  (可调宽 320-760px)     │
│          │                        │                         │
│ 项目说明  │  # 第003章 初入宗门     │  ┌─────────────────┐   │
│ 设定/    │                        │  │ 助手 | 你         │   │
│ 数据层/  │  萧炎站在宗门大殿前，   │  │                 │   │
│ 大纲/    │  心中涌起一股复杂的     │  │ 这段打斗能不能   │   │
│  总大纲   │  情绪...               │  │ 更激烈一点？     │   │
│  开篇设计 │                        │  │                 │   │
│ 正文/    │                        │  │ 好的，改成了：   │   │
│  第001章 │                        │  │ [差异预览]       │   │
│  第002章 │                        │  │ [应用] [放弃]    │   │
│  第003章 │ ●                      │  └─────────────────┘   │
│ 时间轴/  │                        │                         │
│ 附录/    │                        │  模型: GPT-4o | 已连接  │
│ 校验/    │                        │  [Skill] [上下文] [写入] │
│ 导出/    │                        │  [输入框...]      [发送] │
│          │                        │                         │
│          │ 字数: 3247 | 已保存     │                         │
└──────────┴────────────────────────┴─────────────────────────┘
```

#### 3.3.1 核心三栏

**左侧：文档树**

按类别组织的项目文档树，与现有设计一致：

```
📄 项目说明.md
📁 设定/
📁 数据层/         ← JSON，可视化关系图
📁 大纲/
   📄 总大纲.md     ← canonical，可打开编辑
   📄 开篇设计.md   ← canonical，可打开编辑
📁 正文/
   📄 第001章.md
   📄 第002章.md
   📄 第003章.md    ← 当前
📁 时间轴/
📁 附录/
📁 校验/
📁 导出/
```

右键菜单：新建文件/文件夹、重命名、删除、导出。stale 章节显示橙色标记。正在生成中的章节显示加载动画。

**中间：编辑器**

根据文件类型自动切换编辑体验：

| 文件类型 | 编辑器 | 特殊能力 |
|---|---|---|
| `.md` 章节正文 | Markdown 编辑器 | 编辑/分栏/预览三模式、版本历史、字数统计 |
| `.md` 大纲/开篇 | Markdown 编辑器 | 同上 + stale 提示 + 批准操作 |
| `.md` 设定/说明 | Markdown 编辑器 | 自由写作 |
| `.json` 数据层 | JSON 编辑器 | 编辑/关系图可视化 |

编辑器右上角显示保存状态：已保存/未保存/保存中/助手正在写入。

**右侧：AI 共创助手**

这是工作室的灵魂。不是"附加功能"，是创作方式本身。

- **对话式交互**：你想写什么、怎么改、卡住了——跟 AI 聊。
- **上下文感知**：当前打开的文档自动带入对话上下文。
- **工具能力**：AI 可以读文档、写文档、搜索章节、生成大纲——你在聊天里让它做就行。
- **结果直接可用**：AI 返回的 Markdown 内容可以一键应用到当前文档、追加到末尾、或创建新文档。
- **Skill/模型选择**：发送框上方可切换 Skill、选择模型、控制是否允许写入。

聊天面板可拖拽调整宽度（320-760px）。不需要时可以折叠，编辑器自动撑满。

#### 3.3.2 被工作室吸收的能力

以下旧设计中作为独立页面的功能，全部在工作室内部变成**面板、对话框或 AI 对话能力**：

| 旧设计的独立页面 | 在工作室里的形态 |
|---|---|
| 创作准备页（设定/大纲/开篇） | 文档树里的文件。打开即编辑。AI 聊天里让它生成。 |
| 项目总览页 | 不需要。进入工作室就是你上次的位置。顶部面包屑旁可显示关键状态（工作流运行中/有 stale 章节）。 |
| 质量与连续性页 | 融入工作室四层可见性：文档树标记 → 编辑器内标注 → 审校面板汇总 → AI 聊天解释。详见 [§3.3.4](#334-质量连续性四层可见性)。 |
| 导出页 | 文档树右键 → 导出，或工具栏导出按钮 → 弹出范围/格式/版本选择对话框。导出记录在文档树 `导出/` 目录下可查看和下载。 |

#### 3.3.3 编辑器视图模式

为满足不同创作场景，编辑器支持模式切换：

| 模式 | 用途 | 快捷键 |
|---|---|---|
| **写作模式**（默认） | 全屏编辑器 + 侧边栏折叠。沉浸式写作。 | `Ctrl+1` |
| **协作模式** | 编辑器 + AI 聊天并排。边写边聊。 | `Ctrl+2` |
| **审校模式** | 编辑器 + 审校面板（版本历史/审核 issue/Story Bible 冲突）。 | `Ctrl+3` |
| **阅读模式** | 全屏连续阅读。按章节顺序无缝滚动，无编辑控件。像读书一样读自己的作品。 | `Ctrl+4` |

#### 3.3.4 质量/连续性：四层可见性

一致性问题是 AI 辅助长篇写作的真实痛点。不做独立"质量页"，但也不能让问题沉没在聊天记录里。质量信息通过四个层级递进暴露，从轻到重：

```
第 1 层：文档树标记（无感层）
  ├─ 章节旁橙色圆点 = stale（上游设定/大纲变更后此章需关注）
  ├─ 章节旁黄色圆点 = 有审核 issue（非阻断级别）
  ├─ 章节旁红色圆点 = 有阻断级审核 issue
  └─ 章节旁加载动画 = 引擎正在生成/审核此章

第 2 层：编辑器内标注（即时层）
  ├─ 段落左侧色条（黄/橙/红对应 issue 严重度）
  ├─ hover 显示 issue 摘要
  └─ 点击跳转到审校面板查看详情

第 3 层：审校面板（汇总层）
  ├─ 从编辑器右侧滑出（Ctrl+3 审校模式自动开启）
  ├─ 当前章节的 issue 列表：定位、证据、建议修复、处理状态
  ├─ Story Bible 冲突：potential/confirmed、替代链、是否注入上下文
  ├─ stale 传播链：上游变更 → 影响哪些下游章节
  └─ 批量操作：标记已处理、忽略、跳转编辑、重新生成范围

第 4 层：AI 聊天解释（对话层）
  ├─ "@检查一致性" → AI 逐条解释每个 issue 的前因后果
  ├─ 引擎审核结果推送为聊天消息，可继续追问
  └─ "@帮我修这个冲突" → AI 给出修改方案，确认后应用
```

这四层的关系：**第 1 层让问题不被忽略，第 2 层让问题不被脱离上下文，第 3 层让问题可以集中处理，第 4 层让问题可以被理解和修复。** 用户不需要去一个单独的"质量页"——质量信息始终在工作室里，按需要的深度逐层展开。

### 3.4 工作流引擎 `/workspace/project/[id]/engine`

**定位**：自动化批量执行。让 AI 按预设流程帮你生成章节、审核、精修、导出。

**为什么它仍然是一个独立页面（而不是工作室面板）**：

- 引擎的布局需求跟工作室完全不同：时间线、节点详情、日志、SSE 事件流——需要大面积纵向空间。
- 引擎执行时用户通常处于"监控"心智，不写作。但执行完立刻切回工作室继续写作。
- 引擎结果直接关联到工作室文档（生成的章节直接出现在文档树），两个页面之间的数据流是双向的。

**布局**：

```
┌──────────────────────────────────────────────────────────┐
│ [←工作室] 工作流: 批量生成章节  [▶启动] [⏸暂停] [■取消] │
├────────────┬──────────────────┬──────────────────────────┤
│ 执行列表   │  节点时间线       │  详情面板                 │
│            │                  │                          │
│ 本次执行 ● │  generate_003    │  节点: 生成第003章        │
│ 上次执行   │  ✅ completed    │  输入: outline + prev     │
│ 上上次     │                  │  输出: 章节正文           │
│            │  review_003      │  Token: 3247             │
│            │  🔍 running      │  耗时: 12.3s             │
│            │                  │                          │
│            │  fix_003         │  [日志] [Prompt回放]     │
```

### 3.5 分析实验室 `/workspace/project/[id]/lab`

### 3.6 设置

**产品心智上**：设置是书架/工作室里的侧边面板，不是独立工作空间。用户在主空间里调设置，不跳页。

**工程实现上**：当前代码中设置仍以独立路由存在（`/workspace/lobby/settings`、`/workspace/project/[id]/settings`），这为刷新恢复、深链跳转、未保存保护提供了工程支撑。过渡期保留 route-backed panel 形态，后续逐步将设置 UI 迁入主工作空间的侧边抽屉/面板，路由可保留为别名或用于深链。

左侧设置导航：

```
个人设置
  ├─ 账户信息
  ├─ 模型连接
  ├─ AI 偏好
  ├─ 个人规则
  ├─ Skills
  ├─ Agents
  ├─ Hooks
  ├─ MCP
  └─ 费用概览

项目设置（进入项目后）
  ├─ 基础信息
  ├─ 项目规则
  ├─ AI 偏好
  ├─ Skills
  ├─ MCP
  ├─ 凭证
  └─ 危险操作
```

个人设置从书架页进入。项目设置从工作室顶部菜单进入。两者在同一套 UI 框架下，只是 scope 不同。设置面板上方有 scope 切换指示，清楚显示当前在编辑"全局默认"还是"项目覆盖"。

## 5. 为什么不拆更多页

### 5.1 孵化不需要独立页

孵化是创建项目的过程，1-3 轮对话的时长。做成全屏页面就是打断用户——他还在"想写什么"的模糊阶段，就被丢进一个步骤化向导。更自然的做法：在书架页点"新建"，弹出一个对话面板跟用户聊，聊着聊着项目就建好了，用户甚至不需要知道"孵化"这个概念。

### 5.2 设定/大纲/开篇不需要独立页

它们是内容，不是流程。把它们放在文档树里，用户想编辑就点开，想让 AI 生成就在聊天里说——跟编辑章节没有区别。"先填完设定才能写正文"是工具强加的约束，不是创作的自然规律。很多好故事是边写边发现设定的。

### 5.3 质量检查不需要独立工作空间

一致性问题是 AI 辅助长篇写作的真实痛点。但解决方案不是让用户去一个"质量页"盯着问题列表，而是通过工作室内的四层可见性（[§3.3.4](#334-质量连续性四层可见性)）——文档树标记、编辑器内标注、审校面板、AI 聊天解释——让质量信息按需要的深度逐层展开，融入写作流而不是成为写作之外的一个阶段。

### 5.4 导出不需要独立页

导出是一个操作，不是一个工作空间。选择范围、格式、版本 → 导出下载。一个对话框的事。历史导出记录放在文档树 `导出/` 里可以查看下载。

### 5.5 Assistant 不做独立聊天页

通用聊天工具是 ChatGPT，不是 easyStory。AI 对话绑定项目、绑定文档上下文、绑定当前编辑器——才有意义。脱离创作上下文的独立聊天页反而会让产品定位模糊。

## 7. 路由过渡口径

本文档用"主工作空间"描述产品心智，用"路由/面板/对话框"描述工程实现。一个工作空间在工程上可能对应多条路由，这不矛盾。

### 7.1 哪些能力当前以独立路由存在

| 路由 | 产品心智归属 | 过渡期理由 |
|---|---|---|
| `/workspace/lobby/new` | 书架（创建对话） | 深链跳转、浏览器刷新恢复、孵化步骤间未保存保护 |
| `/workspace/lobby/settings` | 书架（设置面板） | 深链跳转、刷新恢复、表单未保存保护 |
| `/workspace/lobby/templates` | 书架（模板面板） | 同上 |
| `/workspace/lobby/recycle-bin` | 书架（回收站面板） | 同上 |
| `/workspace/project/[id]/settings` | 工作室（设置面板） | 同上 |
| `/workspace/lobby/config-registry` | 管理后台（不暴露给普通用户） | 管理员专属入口 |

### 7.2 演进方向

路由保留，但 UI 逐步收进主工作空间：

- **短期**（当前）：保持 route-backed panel。设置、模板、回收站各自有路由，加载对应全屏视图。
- **中期**：当用户从书架/工作室内打开这些面板时，以侧边抽屉/对话框展示，不跳页；路由作为深链别名，直接访问时仍渲染完整视图。
- **长期**：面板 UI 成为主工作空间内的常驻组件；独立路由退化为 shell + 面板组件的组合渲染，或保留用于外部深链。

关键原则：**路由可以多，工作空间不能多。** 工程上的路由拆分是手段，产品上的空间数量是体验。

## 8. 与当前实现的对照

当前 web 应用的核心布局已经走在了正确的方向上：

| 当前实现 | 本文档建议 | 状态 |
|---|---|---|
| Lobby 书架页 | 书架工作空间（增强创建体验） | 核心一致 |
| Studio 工作室 | 工作室（增加阅读模式、审校面板、导出对话框） | 核心布局一致，`studio-document-support.ts` 已支持文档树统一视图 |
| Engine 引擎 | 工作流引擎 | 一致 |
| Lab 实验室 | 分析实验室 | 一致 |
| Incubator 孵化页（独立路由） | 融入书架创建流程（route-backed dialog） | 产品心智收敛，路由短期保留 |
| Settings 设置页（独立路由） | 融入书架/工作室侧边面板（route-backed panel） | 产品心智收敛，路由短期保留 |
| Templates 模板页 | 融入书架侧边面板 | 同上 |
| ConfigRegistry 配置台 | 管理员隐藏入口 | 一致 |

核心调整方向：
1. **孵化**：从独立全屏页收敛为书架内的对话式创建。路由短期保留用于深链/刷新。
2. **设置/模板/回收站**：从独立全屏页收敛为主工作空间内的面板。路由短期保留，UI 逐步迁入。
3. **质量/连续性**：从"无独立承载"升级为工作室内的四层可见性（[§3.3.4](#334-质量连续性四层可见性)）。
4. **导出**：从引擎页专属操作扩展为工作室内的直接可用的对话框。
