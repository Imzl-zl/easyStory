# 09 - 错误处理与容错机制

| 文档类型 | 设计规格 |

| 优先级 | MVP 必须 |

---

## 1. 概述

LLM 调用是不稳定的 I/O 操作，网络超时、API 限流、模型输出异常等问题不可避免。easyStory 必须具备完善的容错机制，确保工作流在遇到错误时能够合理应对，而非直接崩溃。

本文档定义三层容错策略：
1. **错误分类** - 识别错误类型，选择对应处理方式
2. **重试策略** - 对瞬时错误进行自动重试
3. **模型降级** - 当主模型不可用时，自动切换备选模型

---

## 2. 错误分类与处理策略

### 2.1 错误类型表

| 错误类型 | 示例 | 处理策略 |
|---------|------|---------|
| 瞬时错误 | 网络超时、API 限流 429 | 指数退避重试（1s -> 2s -> 4s），最多 3 次 |
| 模型错误 | 输出格式异常、内容为空 | 重新生成，换 prompt 策略 |
| 配置错误 | Skill 不存在、变量缺失 | 立即失败，提示用户修复配置 |
| 预算错误 | Token 超限 | 暂停工作流，通知用户 |
| 内容安全 | 生成了违规内容 | 标记并跳过，记录日志 |

### 2.2 处理原则

- **瞬时错误**: 可重试，重试次数有限，超限后上报
- **模型错误**: 可重试但需要调整策略（如换 prompt），不做简单重复
- **配置错误**: 不可重试，属于用户侧问题，立即终止并给出修复指引
- **预算错误**: 不可重试，需要用户介入（调整预算或确认继续）
- **内容安全**: 不可重试，跳过当前节点并记录，不阻塞整体流程

## 3. 重试策略配置

### 3.1 配置格式

```yaml
workflow:
  retry:
    strategy: "exponential_backoff"  # exponential_backoff / fixed / none
    initial_delay: 1                  # 初始延迟（秒）
    max_delay: 30                     # 最大延迟（秒）
    max_attempts: 3                   # 最大重试次数
    retryable_errors:                 # 可重试的错误类型
      - "timeout"
      - "rate_limit"
      - "server_error"
```

### 3.2 重试策略说明

| 策略 | 延迟模式 | 适用场景 |
|------|---------|---------|
| `exponential_backoff` | 1s -> 2s -> 4s -> ... (封顶 `max_delay`) | 默认推荐，适合 API 限流和网络波动 |
| `fixed` | 固定间隔 `initial_delay` | 已知固定恢复时间的场景 |
| `none` | 不重试 | 调试模式或严格模式 |

### 3.3 可重试错误白名单

只有在 `retryable_errors` 列表中的错误类型才会触发重试。默认包含：
- `timeout` - 网络超时
- `rate_limit` - API 限流（HTTP 429）
- `server_error` - 服务端错误（HTTP 5xx）

配置错误、预算错误等不在白名单中，不会被重试。

## 4. 模型降级策略

### 4.1 降级链配置

仅当工作流**显式启用** `model_fallback` 时，运行时才允许按预设链路切换到备选模型；默认应直接暴露失败并暂停/终止，而不是隐式换模型继续执行。

```yaml
workflow:
  model_fallback:
    enabled: true
    chain:
      - model: "claude-sonnet-4-20250514"    # 首选
      - model: "gpt-4o"              # 降级 1
      - model: "deepseek-v3"         # 降级 2
    on_all_fail: "pause"             # 全部失败时暂停
```

### 4.2 降级触发条件

模型降级在以下情况触发：
- 当前模型的重试次数耗尽（`max_attempts` 次均失败）
- 当前模型返回不可重试的错误（如认证失败、模型下线）
- 当前模型的凭证无效或过期

### 4.3 降级行为

1. 按 `chain` 列表顺序尝试下一个模型
2. 每个降级模型仍然遵循重试策略（拥有独立的重试计数）
3. 降级前必须做能力兼容性检查
4. 如果所有兼容模型均失败，执行 `on_all_fail` 策略

### 4.4 模型能力兼容性检查

并非所有备选模型都能安全替代当前模型。降级时必须校验当前 Skill / Agent / Node 对模型能力的要求。

建议的能力标签：

- `json_schema_output`：稳定输出结构化 JSON / Schema
- `long_context`：支持大上下文
- `streaming`：支持流式输出
- `tool_calling`：支持工具调用

示例：
- reviewer Agent 默认要求 `json_schema_output`
- 长章节生成可能要求 `long_context`
- Studio 流式生成要求 `streaming`

**规则：**
- 若备选模型不满足必需能力，不能进入降级链尝试
- 若当前节点存在能力要求但所有备选模型都不满足，则直接执行 `on_all_fail`
- 禁止为了“降级成功”而静默放宽结构化输出要求

### 4.5 `on_all_fail` 策略

| 策略 | 行为 |
|------|------|
| `pause` | 暂停工作流，等待用户介入（默认） |
| `fail` | 终止工作流，标记为失败 |

> 最佳实践：`model_fallback` 只解决“同一节点的模型可用性”问题，不解决“节点可以被跳过”问题。生成/审核节点在所有候选模型都失败时，不允许因为切换链耗尽而自动 `skip`。

### 4.6 注意事项

- 降级时需要确认备选模型有可用的凭证（参见 10-user-and-credentials.md）
- 不同模型的 token 计价不同，降级后费用可能变化（参见 08-cost-and-safety.md）
- 降级事件应记录日志，便于用户了解实际使用了哪个模型
