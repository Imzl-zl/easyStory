## 1. 决策目标

本次技术决策服务于 easyStory 第一阶段最小闭环，目标只有一个：在尽量适配大多数用户和大多数技术团队的前提下，把“节点式创作 + 人工审核 + 版本追踪 + 导出”做稳。

## 2. 决策范围

本次决策覆盖以下能力：

- Web 工作台
- 后端 API 与业务服务
- 节点工作流编排
- 模型接入抽象
- 数据持久化
- 单机开发与单机部署

本次决策不要求首版同时落地以下能力：

- 插件市场
- 模板市场
- 开放式插件执行
- MCP Server
- 向量检索
- 团队协作
- Kubernetes

## 3. 最终技术栈

### 3.1 核心选型

| 层级 | 组件 | 版本线 | 结论 | 主要原因 |
|---|---|---|---|---|
| 后端运行时 | Python | 3.12+ | 选用 | AI 生态成熟、稳定性和兼容性均衡,3.12 性能更好 |
| API 框架 | FastAPI | 0.115+ | 选用 | 类型清晰、异步友好、内置流式支持和文件上传 |
| 数据验证 | Pydantic | 2.x | 选用 | 适合复杂输入输出结构建模 |
| ORM | SQLAlchemy | 2.0.x | 选用 | Python 主流 ORM，适合后续从 SQLite 迁移到 PostgreSQL |
| 工作流编排 | LangGraph | 0.2.70+ | 选用 | 最贴近节点式审核工作流,支持流式输出和状态管理 |
| 模型接入 | LiteLLM | 1.82+ | 选用 | 统一模型供应商调用方式,MVP 阶段足够 |
| 前端运行时 | Node.js | 24.x Active LTS | 选用 | 当前主流稳定线，适合新项目 |
| 前端框架 | Next.js | 16.x | 选用 | 兼顾工作台、公开页面和未来扩展 |
| UI 框架 | React | 19.2.x | 选用 | 生态成熟，重交互工作台适配度高 |
| 样式体系 | Tailwind CSS | 4.x | 选用 | 构建复杂工作台效率高 |
| 类型系统 | TypeScript | 5.x | 选用 | 提升前端配置和状态复杂度下的可维护性 |
| 远程数据状态 | TanStack Query | 5.x | 选用 | 节点状态轮询、缓存和失效控制更清晰 |
| 本地状态 | Zustand | 5.x | 选用 | 轻量，适合工作台交互状态 |
| 表单 | React Hook Form | 7.x | 选用 | 模型配置和节点配置表单更易维护 |
| 富文本编辑 | TipTap | 2.x | 选用 | 适合章节编辑、局部精修和扩展 |
| 流程图 | React Flow | 12.x | 选用 | 适合工作流节点可视化 |
| 开发数据库 | SQLite | 3.x | 选用 | 快速启动，便于本地开发和演示 |
| 部署数据库 | PostgreSQL | 16.x | 选用 | 正式部署稳定可靠，适合演进 |

### 3.2 架构补充决策

| 主题 | 决策 |
|---|---|
| 前后端关系 | 前端和后端分离，前端负责工作台交互，后端负责业务编排与持久化 |
| 模型调用边界 | 前端不直接调用模型供应商，统一由后端经 LiteLLM 服务层出站 |
| 节点状态同步 | 第一阶段优先采用轮询或 SSE；协作型 WebSocket 延后 |
| 导出策略 | 第一阶段先保证 Markdown，`DOCX/PDF` 作为增强能力追加 |
| 部署方式 | 默认本机开发；部署默认 Docker Compose + Nginx / Caddy |

## 4. 为什么这样选

### 4.1 从产品需求看

easyStory 的核心不是聊天，而是“状态化创作流程”。因此，技术栈必须支持：

- 节点状态流转
- 可恢复执行
- 中间结果审核
- 版本快照留存
- 多模型切换
- 复杂工作台交互

这就决定了它不适合只靠一个聊天前端加少量脚本拼起来。

### 4.2 从大多数用户看

我们选择这套技术栈的一个重要原因，是它更容易服务“大多数真正会使用这个系统的人”。

| 维度 | 决策 | 带来的好处 |
|---|---|---|
| 使用方式 | Web 优先 | 用户不用安装复杂客户端 |
| 上手成本 | 默认模板 + 默认流程 | 新手可以直接开始 |
| 模型选择 | LiteLLM 统一接口 | 用户不被单一供应商锁死 |
| 自托管门槛 | Docker Compose | 小团队和个人更容易部署 |
| 开发门槛 | React + FastAPI + PostgreSQL | 普通技术团队更容易接手 |
| 扩展路径 | 保留模板、Hook 和后续平台化空间 | 既不把首版做太重，也不堵死演进路线 |

### 4.3 从工程实现看

- `FastAPI + Pydantic + SQLAlchemy` 非常适合做结构清晰的业务后端。
- `Next.js + React + TipTap + React Flow` 非常适合做重交互工作台。
- `LangGraph` 负责流程编排，比自研状态机更省时间。
- `LiteLLM` 负责模型抽象，比业务层直连多家供应商更可控。

## 5. 面向大多数用户的适配策略

这一节不是产品口号，而是技术决策的一部分。

### 5.1 对产品用户的适配

- 默认从浏览器进入系统。
- 默认提供可直接使用的创作模板。
- 默认提供一条能跑通的创作流程。
- 不要求用户先懂 Prompt 工程、插件系统或工作流 DSL。

### 5.2 对技术团队的适配

- 使用主流技术栈，降低维护成本。
- 使用主流数据库，降低迁移风险。
- 使用主流部署方式，降低自托管门槛。

### 5.3 对模型生态的适配

第一阶段默认面向以下类型的模型接入：

- OpenAI 兼容接口
- Anthropic Claude
- Google Gemini
- 本地模型扩展路径，例如 Ollama

这不代表第一阶段要全部深度打透，而是代表架构必须能容纳它们。

## 6. 不纳入 MVP 的能力

| 能力 | 处理方式 | 原因 |
|---|---|---|
| Redis | 延期 | 当前没有明显的多实例同步或后台任务压力 |
| 后台任务队列 | 延期 | 第一阶段可先用同步执行或轻量后台方案 |
| MCP Python SDK / MCP Server | 延期 | 不是 web 产品闭环的前提条件 |
| pluggy | 延期 | 扩展点还未稳定，过早引入只会放大复杂度 |
| one-api / new-api | 延期 | 属于网关平台化需求，不是首版必需 |
| 向量库 | 延期 | 目前没有成型的知识检索需求闭环 |
| Kubernetes | 延期 | 当前部署规模不需要 |
| Docker Swarm | 不采用 | 收益不足，不作为推荐方案 |

### 6.1 第一阶段实施约束

为了确保后续设计和开发不偏离“适配大多数用户”的原则，第一阶段默认遵循以下实施约束：

| 约束 | 默认策略 | 说明 |
|---|---|---|
| 部署拓扑 | 单机优先 | 默认支持本机开发和单机部署，不按分布式系统设计 |
| 数据库形态 | 单主数据库 | 首版默认一个业务主库，不拆分多库多存储 |
| 租户模型 | 单用户项目视角 | 首版不按复杂多租户 SaaS 管理后台建模 |
| 模型调用链路 | 后端统一出站 | 前端不直接调用模型供应商 |
| 扩展机制 | 受控配置扩展 | 首版不支持开放式第三方插件执行 |
| 后台任务 | 轻量优先 | 不以消息队列和分布式任务系统为前提 |
| 检索能力 | 不引入 RAG | 没有明确知识检索闭环前，不上向量库 |
| 协作模型 | 单人创作为主 | 不以多人实时协同为首版设计中心 |

这些约束不是永久限制，而是第一阶段的交付边界。只有当默认创作闭环被验证稳定后，才考虑逐项放开。

## 7. 版本治理策略

### 7.1 文档写法

- 正式文档只写版本线，例如 `FastAPI 0.13x`、`Next.js 16.x`。
- 文档中禁止使用 `latest` 作为正式选型结果。

### 7.2 工程落地规则

- 实际项目中必须锁定具体补丁版本。
- Python 依赖通过锁文件管理。
- Node 依赖通过 `packageManager` 和锁文件管理。

### 7.3 升级策略

- 补丁版本按月评估，可滚动升级。
- 主版本升级必须单独记录决策理由。
- 安全补丁优先于功能升级。

### 7.4 时间口径

本文件中的版本线判断基于 2026-03-13 的官方公开信息：

- `Node.js 24` 为 Active LTS。
- `React 19.2` 已稳定发布。
- `Next.js 16` 已稳定发布。
- `Tailwind CSS 4` 已稳定发布。
- `FastAPI`、`LangGraph`、`LiteLLM` 均有稳定主线可用。

## 8. 推荐工程边界

### 8.1 后端边界

- `api` 层负责 HTTP、序列化、错误输出。
- `application` 层负责节点运行、审核动作、导出流程。
- `domain` 层负责项目、节点、版本和审核等核心概念。
- `infrastructure` 层负责数据库、LiteLLM、导出和存储。

### 8.2 前端边界

- 页面层负责路由和页面装配。
- 工作台层负责创作、审核和版本交互。
- 共享组件层负责编辑器、节点卡片、审核面板等复用部件。
- 远程状态与本地交互状态分离，避免全部堆进全局 store。

### 8.3 目录建议

```text
easyStory/
  docs/
  apps/
    api/
    web/
```

## 10. 当前可直接推进的实施项

在本决策生效后，后续工作应直接围绕以下内容展开：
