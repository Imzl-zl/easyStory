## 1. 调研目标

本调研不是为了追逐“最新最酷”的技术，而是为了回答下面几个更实际的问题：

1. 什么技术组合最适合做一个节点式 AI 小说创作系统。
2. 什么技术组合最容易被普通开发者、小团队和自托管用户接受。
3. 什么技术组合能够支撑后续扩展，但不会把第一阶段做得过重。
4. 什么技术组合能兼顾“默认可用”和“后续可扩展”。

## 2. 调研原则

### 2.1 需求匹配优先

技术选型必须围绕核心需求：

- 节点式创作
- 人工审核闭环
- 长文本版本管理
- 多模型接入
- Web 工作台交互

### 2.2 适配大多数用户优先

这里说的“大多数用户”包括两层：

- 产品使用者：新手作者、普通创作者、小型工作室、进阶用户。
- 技术使用者：普通前后端工程师、小团队、自托管用户。

因此技术路线应尽量满足：

- 常见浏览器即可访问。
- 常见开发栈即可维护。
- 默认部署路径简单。
- 不强依赖某一个模型供应商。

### 2.3 先做最小闭环，再做平台化

第一阶段的目标是把创作闭环跑通，而不是一次性做成生态平台。因此：

- `MCP`
- 插件系统
- 向量检索
- Redis / 复杂任务系统
- Kubernetes

都应按“后续能力”看待，而不是“一期必选”。

### 2.4 版本治理优先于追新

文档里不能再用 `latest` 作为正式结论。技术栈必须记录“版本线”，真实项目要锁定补丁版本。

## 3. 系统架构需要什么能力

从系统需求反推，easyStory 的技术栈至少需要提供以下能力：

| 能力 | 原因 |
|---|---|
| Web UI 工作台 | 创作、审核、版本切换和配置都需要图形化界面 |
| 强类型 API | 节点输入输出和状态管理复杂，接口需要清晰定义 |
| 工作流编排 | 节点执行、审核、回退、重试都需要流程控制 |
| 多模型抽象 | 避免系统绑死在单一模型供应商 |
| 持久化和版本管理 | 项目、节点、产物和审核动作都需要可追溯 |
| 可恢复执行 | 节点执行中断后需要恢复 |
| 可渐进扩展 | 后续要有模板、扩展点和平台化空间 |

### 3.1 统一评估维度

为了避免技术选型只停留在“感觉不错”，后续比较统一按以下维度判断：

| 维度 | 关注点 | 为什么重要 |
|---|---|---|
| 需求匹配度 | 是否天然支持节点式创作、审核、版本管理 | 不匹配需求的技术，再热门也会增加补丁代码 |
| 学习成本 | 普通前后端工程师和 AI 工程师是否容易接手 | easyStory 不能依赖极少数专家维护 |
| 部署复杂度 | 本地开发和单机部署是否容易启动 | 适配个人用户和小团队，自托管门槛不能太高 |
| 生态成熟度 | 社区活跃度、文档、示例、第三方集成是否足够 | 直接影响交付速度和长期维护成本 |
| 供应商锁定风险 | 是否容易被某个框架、平台或模型供应商绑死 | 关系到后续模型切换和系统演进 |
| 演进空间 | 首版能否先轻量落地，后续再扩展 | 不能为了未来可能性压垮当前 MVP |

### 3.2 适配大多数用户的技术判断标准

技术上所谓“适配大多数用户”，至少要同时满足下面这些条件：

- 产品用户可以通过浏览器访问，不必先配置复杂本地环境。
- 技术团队可以用主流技术栈维护，不依赖冷门框架。
- 小团队或个人可以在单机环境部署，不必一开始就上复杂编排系统。
- 模型接入不被单一供应商绑定，后续能切换和扩展。
- 首版功能能在合理复杂度内交付，而不是为了远期平台化牺牲当前可用性。

## 4. 前端技术调研

### 4.1 候选方案

| 方案 | 优点 | 风险 | 适配大多数用户的表现 |
|---|---|---|---|
| Next.js 16 + React 19 | 路由、布局、全栈整合、生态丰富 | 对纯内部工作台略重，升级节奏快 | 很强，适合未来有公开页面和工作台并存 |
| Vite + React 19 | 简单、启动快、概念少 | 未来要补更多路由和服务端能力 | 很强，适合纯工作台 |
| Nuxt / Vue | 成熟稳定 | 与 React Flow、TipTap、现成 AI UI 生态相比优势不明显 | 中等 |

### 4.2 结论

- 如果 easyStory 长期只是一套登录后的工作台，`Vite + React` 完全可行。
- 如果 easyStory 未来会同时存在官网、模板展示页、公开作品页和工作台，`Next.js` 更合适。
- 当前更推荐 `Next.js`，理由不是单纯 `SEO`，而是它能把页面路由、服务端能力和工作台放到同一技术体系里。

### 4.3 组件层选择

| 能力 | 方案 | 原因 |
|---|---|---|
| UI 组件 | shadcn/ui 风格组件方案 | 可定制、适合自定义产品界面 |
| 样式 | Tailwind CSS 4.x | 快速搭建复杂工作台界面 |
| 表单 | React Hook Form 7.x | 适合复杂配置表单 |
| 远程数据 | TanStack Query 5.x | 适合节点状态轮询和缓存 |
| 本地状态 | Zustand 5.x | 简单轻量，适合交互状态 |
| 富文本 | TipTap 2.x | 适合章节内容编辑和局部精修 |
| 流程图 | React Flow 12.x | 适合节点流程可视化 |

### 4.4 为什么这套前端更适合大多数用户

- 使用浏览器即可访问，不要求安装桌面客户端。
- React 生态丰富，普通前端开发者更容易接手。
- 富文本、流程图、表单这些重交互需求都能找到成熟方案。

## 5. 后端技术调研

### 5.1 候选方案

| 方案 | 优点 | 风险 | 结论 |
|---|---|---|---|
| FastAPI | 异步友好、OpenAPI 自带、类型清晰、AI 应用生态强 | 需要自己约束工程结构 | 推荐 |
| Flask | 简单灵活 | 异步和类型体系弱 | 不选 |
| Django / Django Ninja | 生态完整 | 对当前产品偏重 | 不选 |

### 5.2 结论

easyStory 的后端不是传统 CMS，也不是简单 CRUD 服务，而是“节点执行 + 模型调用 + 审核动作 + 导出”的业务系统。`FastAPI + Pydantic + SQLAlchemy` 是最贴近这个形态的组合。

### 5.3 为什么它适合大多数技术团队

- Python 在 AI 应用领域是主流选择。
- `FastAPI` 对普通后端工程师和 AI 工程师都比较友好。
- OpenAPI 自动生成能力能降低前后端协作成本。

## 6. 工作流引擎调研

### 6.1 候选方案

| 方案 | 优点 | 风险 | 结论 |
|---|---|---|---|
| LangGraph 1.0.x | 状态图、人机介入、分支控制、持久化能力成熟 | 学习成本高 | 推荐 |
| 自研状态机 | 完全可控、概念简单 | 需要自己补很多基础能力 | 备选 |
| CrewAI | 角色协作表达直观 | 对严格节点状态和审核动作支撑较弱 | 不选作主引擎 |

### 6.2 结论

easyStory 的核心不是“多角色对话代理”，而是“节点式审核工作流”。从这个角度看，`LangGraph` 比 `CrewAI` 更匹配。

### 6.3 风险提醒

即便使用 `LangGraph`，也不能把业务模型和持久化结构直接绑死在框架上。框架只负责编排，业务概念必须由系统自己定义。

## 7. 模型接入层调研

### 7.1 为什么需要统一模型抽象

如果系统直接耦合某一家模型供应商，会带来几个问题：

- 用户迁移成本高。
- 不同模型供应商的 API 结构差异会快速扩散到业务代码。
- 后续想支持本地模型和国内外不同供应商时会越来越难维护。

### 7.2 候选方案

| 方案 | 优点 | 风险 | 结论 |
|---|---|---|---|
| LiteLLM SDK / Proxy | 多模型统一接口，迁移成本低 | 增加一层抽象，需要处理差异 | 推荐 |
| 直连供应商 SDK | 简单直接 | 多供应商后维护成本高 | 不推荐长期使用 |
| one-api / new-api | 适合渠道管理和统一配额 | 更偏网关平台，不适合首版 | 延期 |

### 7.3 结论

- MVP 用 `LiteLLM SDK` 即可，不必一开始部署 Proxy。
- 当出现多租户、统一渠道管理、额度控制等需求时，再考虑网关模式。

### 7.4 为什么它更适合大多数用户

- 对产品用户来说，可以更灵活地切换模型。
- 对开发者来说，不必在业务代码里到处写不同供应商适配逻辑。
- 对自托管用户来说，也保留了接本地模型的路径。

## 8. 数据存储与后台能力调研

### 8.1 主数据库

| 方案 | 优点 | 风险 | 结论 |
|---|---|---|---|
| SQLite | 开发和演示启动快、零配置 | 不适合作为正式部署主库 | 开发使用 |
| PostgreSQL 16.x | 稳定、功能强、扩展性好 | 运维要求高于 SQLite | 正式部署默认选择 |
| MySQL | 常见成熟 | 在 JSON、生态和扩展路径上相对一般 | 非首选 |

### 8.2 缓存与任务系统

| 能力 | 第一阶段建议 | 原因 |
|---|---|---|
| Redis | 暂不必选 | 当前主要是单机、单流程、单用户交互闭环 |
| 后台任务队列 | 暂不强制 | 可先从同步执行或轻量后台任务开始 |

### 8.3 向量检索

| 方案 | 第一阶段建议 | 原因 |
|---|---|---|
| 不引入向量库 | 推荐 | 当前需求尚未形成明确 RAG 闭环 |
| pgvector / Qdrant | 二期候选 | 如果后续有知识库、设定库、素材检索再引入 |

### 8.4 结论

easyStory 第一阶段真正需要的是“版本追踪、审计和恢复”，不是为了未来想象空间提前引入复杂检索体系。

## 9. 扩展机制与平台能力调研

### 9.1 当前概念区分

| 概念 | 第一阶段定位 |
|---|---|
| Skill | 作为配置资产存在 |
| Hook | 保留受控扩展点 |
| Plugin | 暂不开放运行时安装 |
| MCP | 二期集成方向，不是首版前提 |

### 9.2 结论

如果第一阶段就做开放插件、MCP 和平台市场，会直接抬高系统复杂度。对大多数用户而言，这些能力不是首要价值。

## 10. 部署方案调研

### 10.1 候选方案

| 场景 | 推荐方案 | 不推荐方案 | 原因 |
|---|---|---|---|
| 本地开发 | 本机运行或 Docker Compose | Kubernetes | 开发效率和上手成本更重要 |
| 单机部署 | Docker Compose + Nginx / Caddy | Docker Swarm | 更简单、更常见 |
| 规模化部署 | 未来再评估 Kubernetes | 过早设计复杂编排 | 当前阶段没有必要 |

### 10.2 为什么它更适合大多数人

- 普通开发者容易启动。
- 小团队容易部署。
- 自托管用户不需要理解复杂容器编排。

## 12. 推荐落地顺序

### 12.1 第一阶段必须落地

- `Next.js + React` 工作台
- `FastAPI + Pydantic + SQLAlchemy`
- `LangGraph` 工作流编排
- `LiteLLM` 模型调用抽象
- `SQLite` 开发、`PostgreSQL` 部署
- 默认模板、节点审核、版本记录、Markdown 导出

### 12.2 第二阶段再考虑

- `Redis`
- 后台任务队列
- `MCP Server`
- 插件系统
- 向量检索
- 对象存储
- 团队协作

## 13. 技术风险与缓解

| 风险 | 说明 | 缓解方式 |
|---|---|---|
| LangGraph 耦合过深 | 容易把业务状态绑进框架对象 | 业务实体和框架编排分离 |
| LiteLLM 差异抽象不完整 | 各模型在工具调用和流式输出上仍有差异 | 在服务层统一返回结构和错误模型 |
| Next.js 升级节奏快 | 大版本和安全补丁更新频繁 | 锁版本线、按月评估补丁升级 |
| SQLite 与 PostgreSQL 差异 | 事务和并发行为不同 | 从一开始就按 PostgreSQL 语义设计 |
| 过早平台化 | 时间花在市场、插件、协议生态上 | 严格执行 MVP 边界 |

## 14. 调研结论

- easyStory 的技术主线没有选偏，核心路线是成立的。
- 真正需要修正的不是技术方向，而是系统复杂度节奏。
- 面向大多数用户时，第一阶段应优先保证“默认流程可跑、审核闭环清晰、部署简单、模型可切换”。
- 最终决策文档应保留主栈，弱化不必要的首版复杂度，并把延期能力明确移出 MVP。
