## ⚠️ 讨论规则

### 讨论范围

**我们讨论的是框架层面的问题**：
- ✅ 配置格式怎么设计（字段、类型、校验规则）
- ✅ 配置怎么加载和执行（机制、流程）
- ✅ 用户怎么自定义配置（扩展方式）
- ✅ 配置文件放在哪里（存储位置）
- ✅ 配置如何发现和注册（加载机制）

**我们不讨论具体实现**：
- ❌ 具体内置哪些 Skills（用户自己定义）
- ❌ 具体内置哪些 Agents（用户自己定义）
- ❌ 具体内置哪些 Hooks（用户自己定义）
- ❌ 具体的提示词怎么写（用户自己定义）
- ❌ 具体的工作流怎么配置（用户自己定义）

### 设计原则

1. **平台思维**：我们做的是平台，不是具体应用
2. **可配置优先**：一切可配置，不要写死
3. **用户自定义**：用户可以自由定义 Skills/Agents/Hooks/Workflows
4. **导入导出**：用户可以导入别人分享的配置
5. **版本管理**：配置应该支持版本管理

## 1. 产品定位与核心概念

### 1.1 产品定位

easyStory 是一个 **AI 小说创作平台**，类似于：
- **Cursor**（AI代码编辑器）→ 可以自定义 rules、MCP 工具
- **Claude Code**（AI CLI工具）→ 可以自定义 commands、hooks
- **Windsurf** → 类似的 AI 工具

**核心特点**：
1. 有核心的 AI 能力
2. 可以通过配置文件自定义行为
3. 可以集成外部工具
4. 可以定义 hooks（在特定时机执行操作）

### 1.2 核心概念解释

#### 什么是"工作流"？

**简单理解**：工作流就是"做事的步骤"。

**例子**：写小说的步骤
```
第1步：输入题材和设定
    ↓
第2步：生成大纲
    ↓
第3步：审核大纲（用户确认）
    ↓
第4步：生成章节
    ↓
第5步：审核章节（用户确认）
    ↓
第6步：导出成稿
```

每个步骤就是一个"节点"。

---

#### 什么是"Skill"？

**简单理解**：Skill 就是"AI 的技能包"。

**例子**：
- `skill.chapter.xuanhuan` → 写玄幻小说章节的技能
- `skill.outline.suspense` → 写悬疑小说大纲的技能
- `skill.character.design` → 设计人物的技能

**一个 Skill 包含**：
```yaml
skill:
  id: "skill.chapter.xuanhuan"
  name: "玄幻章节生成"
  prompt: |
    你是一位玄幻小说作家，请根据以下设定生成章节...
  model: "gpt-4"
  temperature: 0.8
```

**作用**：让 AI 知道"怎么写"某类内容。

---

#### 什么是"Hook"？

**简单理解**：Hook 就是"在某个时机自动执行的钩子"。

**例子**：
- `before_generate` → 生成前执行（比如检查输入）
- `after_generate` → 生成后执行（比如自动保存）
- `on_review` → 审核时执行（比如给出建议）

**一个 Hook 的配置**：
```yaml
hook:
  id: "hook.auto_save"
  trigger: "after_generate"
  action: "save_to_database"
```

**作用**：自动化一些操作，减少手动操作。

---

#### 什么是"Agent"？

**简单理解**：Agent 就是"有特定角色的 AI 助手"。

**例子**：
- `agent.outline_writer` → 专门写大纲的助手
- `agent.chapter_writer` → 专门写章节的助手
- `agent.reviewer` → 专门审核内容的助手

**一个 Agent 的配置**：
```yaml
agent:
  id: "agent.outline_writer"
  name: "大纲撰写专家"
  system_prompt: |
    你是一位经验丰富的小说大纲撰写专家...
  skills:
    - "skill.outline.xuanhuan"
    - "skill.outline.suspense"
```

**作用**：让不同的 AI 助手专注于不同任务。

---

#### 什么是"插件"？

**简单理解**：插件就是"扩展功能的模块"。

**例子**：
- `plugin.export.docx` → 导出为 Word 文档
- `plugin.export.pdf` → 导出为 PDF
- `plugin.check.grammar` → 语法检查

**作用**：按需扩展功能，不用的不加载。

---

#### 什么是"MCP"？

**简单理解**：MCP（Model Context Protocol）就是"让 AI 能调用外部工具的协议"。

**例子**：
- AI 可以调用"搜索工具"查找资料
- AI 可以调用"数据库工具"读取设定
- AI 可以调用"文件工具"保存内容

**作用**：让 AI 不只是"聊天"，还能"做事"。

## 3. 核心功能讨论

### 3.1 自动化工作流 ✅ 已讨论

**用户需求**：
> 我可能设置的很好，就能自动化，比如我生成大纲，章节之后，可以hooks 多个agents 审查文风/违禁词/AI味等等用户可以自定义这是灵活的，然后就是用户可以设置hooks 什么时候调用agents 进行审核，精修什么的，然后做一套工作流，自动产出。

**实现方案**：

```
生成大纲 → 自动审核 → 自动精修 → 自动生成章节 → 自动审核 → 自动精修 → ... → 导出
```

**配置示例**：
```yaml
workflow:
  name: "玄幻小说自动创作"
  
  nodes:
    - id: "outline"
      type: "generate"
      skill: "skill.outline.xuanhuan"
      auto_review: true          # 自动审核
      reviewers:                  # 审核 agents
        - agent.style_checker
        - agent.banned_words_checker
        - agent.ai_flavor_checker
      auto_fix: true             # 审核不通过自动精修
      max_fix_attempts: 3        # 最多精修3次
      on_fix_fail: "pause"       # 精修失败后暂停，等待用户
      
    - id: "chapter_plan"
      type: "generate"
      skill: "skill.chapter_plan"
      auto_review: true
      reviewers:
        - agent.consistency_checker
      
    - id: "chapter_gen"
      type: "generate"
      skill: "skill.chapter.xuanhuan"
      loop: true                 # 循环生成每一章
      auto_review: true
      reviewers:
        - agent.style_checker
        - agent.banned_words_checker
        - agent.ai_flavor_checker
        - agent.plot_consistency_checker
      auto_fix: true
      max_fix_attempts: 2
      
    - id: "export"
      type: "export"
      format: "markdown"
```

**流程图**：
```
┌─────────────┐
│ 生成大纲    │
└──────┬──────┘
       │
       ▼
┌─────────────────────────────────────────────┐
│          自动审核（并行执行）                 │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐       │
│  │文风检查 │ │违禁词  │ │AI味检查│       │
│  └────┬────┘ └────┬────┘ └────┬────┘       │
│       └───────────┼───────────┘             │
│                   ▼                         │
│           ┌───────────────┐                 │
│           │ 汇总审核结果  │                 │
│           └───────┬───────┘                 │
└───────────────────┼─────────────────────────┘
                    │
           ┌────────┴────────┐
           ▼                 ▼
      [全部通过]        [有问题]
           │                 │
           ▼                 ▼
    ┌──────────────┐  ┌──────────────┐
    │ 进入下一节点 │  │ 自动精修     │
    └──────────────┘  └──────┬───────┘
                              │
                              ▼
                      ┌───────────────┐
                      │ 重新审核      │
                      │ (最多3次)     │
                      └───────┬───────┘
                              │
                     ┌────────┴────────┐
                     ▼                 ▼
                [通过]            [失败]
                     │                 │
                     ▼                 ▼
              进入下一节点      暂停，等待用户
```

**决策**：✅ 第一阶段实现

### 3.2 上下文注入 ✅ 已讨论

**用户需求**：
> 为了避免上下文爆炸，也为了避免脱离剧情，能不能设置，就是写下一章，要把大纲，目录，前两章先注入，然后再生成下一章，以后每一章都这样，可以自定义设置，注入什么？

**实现方案**：

```
写第3章时注入：大纲 + 目录 + 第1章 + 第2章
写第4章时注入：大纲 + 目录 + 第2章 + 第3章
...
```

**配置示例**：
```yaml
context_injection:
  enabled: true
  
  rules:
    - node: "chapter_gen"
      inject:
        - type: "outline"        # 注入大纲
          required: true
        - type: "chapter_list"   # 注入章节目录
          required: true
        - type: "previous_chapters"  # 注入前N章
          count: 2               # 前2章
          required: true
        - type: "character_profile"  # 注入人物设定（可选）
          required: false
          
  custom_rules:
    - node: "chapter_gen"
      inject:
        - type: "world_setting"  # 注入世界观设定
          required: false
        - type: "previous_chapters"
          count: 5               # 改成前5章
```

**上下文构建流程**：
```
┌─────────────────────────────────────────────────────────────┐
│                    上下文构建器                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 加载大纲                                                │
│     └─→ "这是一个修仙世界，主角叫张三..."                   │
│                                                             │
│  2. 加载章节目录                                            │
│     └─→ "第1章：入门 | 第2章：突破 | 第3章：？"             │
│                                                             │
│  3. 加载前2章内容                                           │
│     └─→ 第1章内容（约3000字）                               │
│     └─→ 第2章内容（约3000字）                               │
│                                                             │
│  4. 组装完整提示词                                          │
│     ┌─────────────────────────────────────────────────┐   │
│     │ 【大纲】                                         │   │
│     │ 这是一个修仙世界...                              │   │
│     │                                                  │   │
│     │ 【章节目录】                                     │   │
│     │ 第1章：入门                                      │   │
│     │ 第2章：突破                                      │   │
│     │ 第3章：？（待生成）                              │   │
```

### 3.3 向量数据库/分词/知识库 ✅ 已讨论

**用户问题**：
> 需不需要向量数据库，分词，知识库什么的？

**分析结论**：

| 功能 | 第一阶段 | 第二阶段 | 难度 | 说明 |
|-----|---------|---------|------|------|
| 向量数据库 | ❌ 不需要 | ✅ 需要 | ⭐⭐⭐⭐ | 第一阶段直接注入即可 |
| 分词 | ❌ 不需要 | ⚠️ 可选 | ⭐⭐ | AI 审核更智能 |
| 知识库 | ❌ 不需要 | ✅ 需要 | ⭐⭐⭐ | 人物库、世界观库 |

**第一阶段不需要向量数据库的原因**：
- 小说创作有明确的上下文结构（大纲、章节、前后文）
- 直接注入前 N 章就能保持连贯性
- 向量数据库主要用于"语义搜索"，我们不需要搜索

**第二阶段可能需要的场景**：
- 人物库：搜索"张三的性格是什么？"
- 世界观库：搜索"这个世界的修炼体系是什么？"
- 素材库：搜索"类似的情节怎么写？"

**决策**：
- ✅ 第一阶段：直接注入方案
- ⬜ 第二阶段：向量数据库 + 知识库

### 3.4 多 Agent 审核 ✅ 已讨论

**框架能力**：

系统支持用户自定义多个审核 Agent，每个 Agent 负责不同的审核维度：

- **文风检查**：检查文风是否符合设定
- **违禁词检查**：检查是否有敏感词
- **AI 味检查**：检查是否有明显的 AI 生成痕迹
- **剧情一致性检查**：检查是否与前后文矛盾
- **设定一致性检查**：检查人物、世界观设定是否一致

**审核流程**：

```
生成内容 → 并行调用多个 Agent → 汇总审核结果 → 决策（通过/精修/暂停）
```

**配置方式**：

用户通过 YAML 配置文件定义审核 Agent，系统在工作流节点中调用这些 Agent 进行审核。

**决策**：✅ 第一阶段实现多 Agent 审核框架

### 9.5 Hook 配置格式

**用途**：定义钩子，在特定时机自动执行操作。

**完整格式**：
```yaml
hook:
  id: "hook.auto_save"                # 必填，唯一标识
  name: "自动保存"                     # 必填，显示名称
  version: "1.0.0"                    # 可选，版本号
  description: "生成后自动保存内容"     # 可选，描述
  author: "user"                      # 可选，作者
  enabled: true                       # 可选，是否启用，默认 true
  
  trigger:                            # 必填，触发条件
    event: "after_generate"           # 触发事件
    node_types:                       # 适用的节点类型（可选）
      - "generate"
      - "fix"
  
  condition:                          # 可选，额外触发条件
    field: "status"                   # 字段名
    operator: "=="                    # 操作符：==/!=/>/</>=/<=/in/not_in
    value: "completed"                # 值
  
  action:                             # 必填，执行动作
    type: "script"                    # 类型：script/webhook/agent
    config:
      module: "app.hooks.builtin"     # 模块路径（script 类型）
      function: "auto_save_content"   # 函数名
      params:                         # 参数
        save_version: true
  
  priority: 10                        # 可选，优先级，数字越小越先执行
  timeout: 30                         # 可选，超时时间（秒）
  retry:                              # 可选，重试配置
    max_attempts: 3
    delay: 1                          # 重试间隔（秒）
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| `id` | string | ✅ | 唯一标识，格式：`hook.{name}` |
| `name` | string | ✅ | 显示名称 |
| `trigger.event` | string | ✅ | 触发事件 |
| `trigger.node_types` | array | ❌ | 适用的节点类型 |
| `condition` | object | ❌ | 额外触发条件 |
| `action.type` | string | ✅ | 动作类型：script/webhook/agent |
| `action.config` | object | ✅ | 动作配置 |
| `priority` | integer | ❌ | 优先级，默认 10 |
| `enabled` | boolean | ❌ | 是否启用，默认 true |

**触发事件列表**：

| 事件 | 说明 | 触发时机 |
|-----|------|---------|
| `before_workflow_start` | 工作流开始前 | 用户点击开始 |
| `after_workflow_end` | 工作流结束后 | 所有节点完成 |
| `before_node_start` | 节点开始前 | 节点执行前 |
| `after_node_end` | 节点结束后 | 节点执行后 |
| `before_generate` | 生成前 | 调用 AI 前 |
| `after_generate` | 生成后 | AI 返回结果后 |
| `before_review` | 审核前 | 开始审核前 |
| `after_review` | 审核后 | 审核完成后 |
| `on_review_fail` | 审核失败 | 审核不通过时 |
| `before_fix` | 精修前 | 开始精修前 |
| `after_fix` | 精修后 | 精修完成后 |
| `on_error` | 错误发生 | 任何错误时 |

**动作类型说明**：

| 类型 | 说明 | config 字段 |
|-----|------|------------|
| `script` | 执行 Python 函数 | `module`, `function`, `params` |
| `webhook` | 调用 HTTP 接口 | `url`, `method`, `headers`, `body` |
| `agent` | 调用 Agent | `agent_id`, `input_mapping` |

### 9.6 Workflow 配置格式

**用途**：定义工作流，包含节点列表和执行规则。

**完整格式**：
```yaml
workflow:
  id: "workflow.xuanhuan_auto"        # 必填，唯一标识
  name: "玄幻小说自动创作"              # 必填，显示名称
  version: "1.0.0"                    # 可选，版本号
  description: "全自动玄幻小说创作流程" # 可选，描述
  author: "user"                      # 可选，作者
  tags: ["玄幻", "全自动"]            # 可选，标签
  
  settings:                           # 可选，全局设置
    auto_proceed: true                # 审核通过后自动进入下一步
    max_retry: 3                      # 最大重试次数
    save_on_step: true                # 每步自动保存
    timeout: 300                      # 单节点超时（秒）
  
  context_injection:                  # 可选，上下文注入规则
    enabled: true
    rules:
      - node_pattern: "chapter_*"     # 匹配章节节点（支持通配符）
        inject:
          - type: "outline"           # 注入类型
            required: true            # 是否必填
          - type: "chapter_list"
            required: true
          - type: "previous_chapters"
            count: 2                  # 前 2 章
            required: true
          - type: "character_profile"
            required: false
  
  nodes:                              # 必填，节点列表
    - id: "outline"                   # 节点 ID
      name: "生成大纲"                 # 节点名称
      type: "generate"                # 节点类型
      skill: "skill.outline.xuanhuan" # 使用的技能
      
      hooks:                          # 节点级钩子
        before:
          - "hook.validate_input"
        after:
          - "hook.auto_save"
      
      auto_review: true               # 是否自动审核
      reviewers:                      # 审核 agents
        - "agent.consistency_checker"
      auto_fix: true                  # 是否自动精修
      max_fix_attempts: 3             # 最大精修次数
      on_fix_fail: "pause"            # 精修失败后动作：pause/skip/fail
    
    - id: "chapter_plan"
      name: "规划章节"
      type: "generate"
      skill: "skill.chapter_plan"
      depends_on: ["outline"]         # 依赖的节点
      
      auto_review: true
      reviewers:
        - "agent.consistency_checker"
    
    - id: "chapter_gen"
      name: "生成章节"
      type: "generate"
      skill: "skill.chapter.xuanhuan"
      depends_on: ["chapter_plan"]
      
      loop:                           # 循环配置
        enabled: true
        count_from: "chapter_plan"    # 从章节规划获取数量
        item_var: "chapter_index"     # 循环变量名
      
      hooks:
        before:
          - "hook.inject_context"
        after:
          - "hook.auto_save"
      
      auto_review: true
      reviewers:
        - "agent.style_checker"
        - "agent.banned_words_checker"
        - "agent.ai_flavor_checker"
        - "agent.plot_consistency_checker"
      auto_fix: true
      max_fix_attempts: 2
      on_fix_fail: "pause"
    
    - id: "export"
      name: "导出成稿"
      type: "export"
      depends_on: ["chapter_gen"]
      
      formats:
        - "markdown"
        - "docx"
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| `id` | string | ✅ | 唯一标识，格式：`workflow.{name}` |
| `name` | string | ✅ | 显示名称 |
| `settings` | object | ❌ | 全局设置 |
| `context_injection` | object | ❌ | 上下文注入规则 |
| `nodes` | array | ✅ | 节点列表 |

**节点类型说明**：

| 类型 | 说明 | 必填字段 |
|-----|------|---------|
| `generate` | 生成内容 | `skill` |
| `review` | 审核内容 | `reviewers` |
| `export` | 导出文件 | `formats` |
| `custom` | 自定义节点 | `action` |

**上下文注入类型**：

| 类型 | 说明 | 参数 |
|-----|------|------|
| `outline` | 大纲 | - |
| `chapter_list` | 章节目录 | - |
| `previous_chapters` | 前 N 章 | `count` |
| `character_profile` | 人物设定 | - |
| `world_setting` | 世界观设定 | - |
| `custom` | 自定义 | `source`, `query` |

### 9.8 配置加载机制

**加载流程**：
```
启动时：
1. 扫描 /config/ 目录
2. 解析所有 YAML 文件
3. 校验配置格式
4. 注册到内存注册表
5. 启动文件监听（热更新）

运行时：
1. Web UI 编辑配置
2. 写入 YAML 文件
3. 文件监听触发
4. 重新加载配置
5. 通知 Web UI 刷新
```

**校验规则**：
- 必填字段检查
- 类型检查
- ID 唯一性检查
- 引用完整性检查（如 skill 引用的 agent 是否存在）

**错误处理**：
- 配置错误：记录日志，跳过该配置
- 引用缺失：记录警告，延迟加载
- 格式错误：返回详细错误信息
