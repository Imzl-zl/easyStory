## 1. 文档定位

本文档只定义当前前端主创作路径的产品心智、页面框架和视觉基线。

不负责：

- 业务状态真值
- API 字段设计
- workflow / content / assistant 的领域规则
- 再写一套与代码脱节的 token 字典

当前 UI token 运行时真值在：

- `apps/web/src/app/globals.css`

## 2. 产品心智

### 2.1 核心判断

easyStory 不是后台管理系统，也不是配置中心。

它是一个小说创作工作台。用户进入后应该先看到：

- 作品
- 当前写作位置
- 下一步创作动作
- 与当前作品直接相关的规则、Skills、模型连接和工具

而不是先看到：

- 资源类型
- 控制面
- 注册表
- JSON 编辑器

### 2.2 配置即产品体验

配置不是后台动作，而是用户塑造自己助手和项目工作方式的过程。

普通用户只保留两层作用域：

- 我的
- 当前项目

用户可见词汇应优先使用：

- 我的助手
- 规则
- Skills
- 模型连接
- 工具

避免把 `Config Registry`、`控制面`、`资源管理` 这类系统词作为主心智暴露给普通用户。

## 3. 主路径信息架构

### 3.1 顶层视图

| 视图 | 定位 | 主职责 |
|---|---|---|
| `Auth` | 登录入口 | 登录、注册 |
| `Lobby` | 项目书架 | 项目入口、模板入口、我的助手入口 |
| `Incubator` | 项目起稿 | 对话起稿、模板起稿、整理项目草稿 |
| `Studio` | 创作桌面 | 设定、大纲、章节、Markdown 写作、右侧协作 |
| `Engine` | 推进控制台 | workflow、章节任务、执行监控 |
| `Lab` | 洞察工作台 | 分析结果、灵感参考 |

### 3.2 一级导航原则

一级导航只服务创作路径，不服务后台心智。

推荐词汇：

- 书架
- 创作
- 推进
- 洞察

以下入口不进入一级主导航：

- 高级配置
- Config Registry
- 低频系统控制页

它们可以存在，但只能作为次级入口或高级入口。

## 4. WorkspaceShell

### 4.1 上下文感知顶栏

同一个产品内，不同页面的用户心智不同，所以顶栏必须切换形态。

#### Lobby 模式

- 显示完整顶栏
- 左侧是品牌文字 `easyStory / 写作空间`
- 中间是创作路径导航：`书架 / 创作 / 推进 / 洞察`
- 右侧是 `我的助手`、用户信息、退出

#### Studio 模式

- 顶栏变成项目级工具栏
- 左侧保留 `返回书架`
- 中间保留项目级路径导航
- 右侧只保留 `项目设置`、用户信息、退出
- 视觉上更轻、更薄，不抢正文

#### Engine / Lab / Project Settings

- 仍使用项目级工具栏
- 保持一致的返回和项目上下文
- 不再回到 Lobby 式的完整导航噪音

## 5. Lobby（书架）

### 5.1 空间角色

Lobby 不是系统首页，而是书架首页。

它应该给用户的感觉是：

- 这是我的作品入口
- 我可以从这里继续写
- 模板和助手都围着作品工作

### 5.2 布局

- 左侧：轻侧栏，承载 `我的作品 / 我的助手 / 模板库 / 回收站`
- 右侧：主舞台，承载标题、搜索、统计卡、项目卡片网格
- 顶部大标题采用更强的编辑/出版感，而不是后台标题栏

### 5.3 项目卡片

项目卡片采用“书卡”而不是“管理卡”表达：

- 顶部色带提供项目识别感
- 标题比按钮更重要
- 优先展示题材、目标字数、模板来源、最近更新时间
- 主 CTA 是 `继续创作`
- 设置是辅助动作，不占视觉主位

---

## 6. Incubator（项目起稿）

### 6.1 空间角色

Incubator 不是新建项目后台，也不是模板表单中心。

它是用户把一个想法整理成“可写项目”的启动空间。

### 6.2 布局

- 顶部使用强叙事感 masthead，先解释“为什么在这里起稿”
- 右侧或上方用模式卡片切换 `AI 聊天 / 模板起稿`
- 下方主舞台只服务当前模式，不并排展示多余控制区
- 反馈信息作为轻提示条，不做后台告警式堆叠

### 6.3 表达原则

- `AI 聊天` 是一起聊故事，不是配置向导
- `模板起稿` 是补齐必要信息，不是后台建表
- 文案优先使用 `起稿 / 草稿 / 当前作品`，避免 `资源 / 配置 / 控制面`

---

## 7. Studio（创作桌面）

### 6.1 总体原则

Studio 必须像写作桌面，而不是三块后台面板并排。

核心原则：

- 正文永远是主舞台
- 三栏之间不用硬分隔线
- 靠背景色和留白自然分层
- 右侧助手可以收起

### 6.2 三栏结构

| 区域 | 宽度策略 | 角色 |
|---|---|---|
| 左栏 | 220-300px | 创作结构：设定 / 大纲 / 正文 |
| 中栏 | 弹性主舞台 | Markdown 写作与预览 |
| 右栏 | 304-404px | 共创助手与上下文协作 |

### 6.3 视觉语言

- 左栏背景：暖灰米色，像放置资料的桌面区域
- 中栏背景：暖白纸面，正文密度最低
- 右栏背景：更浅的暖米色，作为协作侧板
- 无 1px 强边框切割三栏
- 编辑区内边距明显大于侧栏

### 6.4 编辑器

编辑器应接近“纸页”感：

- 标题使用更有编辑感的 serif 标题字
- 正文行高偏大，阅读压力低
- 文稿内容列宽收口，不拉满整屏
- 预览与编辑共享同一纸面基调
- 底部状态栏极简，只显示字数与保存状态

### 6.5 助手面板

右侧面板是共创入口，不是系统控制台。

当前阶段：

- 支持上下文勾选
- 支持对话入口
- 支持复制 / 追加到文档 / 新建文档动作
- 必须明确仍在接入阶段，不伪装为正式生成能力

---

## 8. Engine（推进）

### 8.1 空间角色

Engine 是作品推进室，不是 workflow 控制台首页。

用户在这里应优先感知：

- 当前推进到哪里
- 是否被卡住
- 下一步该继续、暂停还是回到创作

### 8.2 布局

- 顶部使用推进概览 + 行动卡片，不再是通用页头控件区
- 左侧是状态与调试侧栏，右侧是执行详情主舞台
- `执行批次` 可以保留，但只作为继续历史推进的辅助入口，不做页面视觉主角

---

## 9. Lab（洞察）

### 9.1 空间角色

Lab 是围绕作品的研究桌，不是分析记录管理页。

### 9.2 布局

- 顶部先说明洞察对作品的作用，再展示当前焦点和筛选状态
- 主体采用 `洞察列表 / 当前焦点 / 新建洞察` 三栏或堆叠结构
- 技术字段要翻译成用户语义，例如：
  - `content_id` -> `关联正文片段`
  - `generated_skill_key` -> `来源 Skill`

## 10. 视觉基线

### 7.1 风格关键词

- 温暖
- 克制
- 呼吸感
- 编辑感
- 上下文感知

### 7.2 视觉约束

- 避免大圆角
- 避免重阴影
- 避免满屏按钮胶囊和强控件感
- 避免把系统操作按钮放到正文同一视觉层级
- 避免紫色渐变、模板味 landing page 语言

### 7.3 排版约束

- 标题允许使用 serif 字体增强作品感
- 正文仍以高可读性 sans / serif 组合为主
- 重要数字使用 tabular-nums
- 保持 Linear 式的精确对齐和紧凑控件节奏

### 7.4 响应式

- 大屏优先保持三栏
- 中屏允许收窄助手栏
- 小屏按 `目录 -> 正文 -> 助手` 垂直堆叠
- 不能靠隐藏正文核心信息来适配移动端
