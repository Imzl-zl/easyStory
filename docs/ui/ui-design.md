## 1. 文档定位

| 文档 | 负责内容 | 不负责内容 |
|---|---|---|
| `ui-design.md` | 顶层 IA、主视图边界、空间气质、视觉 token | 接口细节、状态映射、边界态规则 |
| `ui-interaction-supplement.md` | 交互流程、状态口径、响应式、无障碍、异常态 | 重新定义产品能力或后端真值 |

### 1.1 能力标记

| 标记 | 含义 |
|---|---|
| `MVP 已支持` | 当前后端已有接口或明确数据真值，前端可直接实施 |
| `MVP UI 占位` | 可以保留入口或只读占位，但完成闭环仍需后端继续补口 |
| `Future` | 二期能力，本轮不作为前端实施真值 |

### 1.2 总体原则

- 顶层主视图稳定为 `Lobby / Studio / Engine / Lab`，避免继续扩张平级页面。
- 复杂功能优先以抽屉、侧板、对话框、子视图承载，不把所有能力都做成新页面。
- 视觉语言保留“水墨流”方向，但必须服务可读性、可操作性和实现成本。
- 当前阶段以大屏 Web 工作台优先，移动端只承担浏览和轻操作。

## 3. 视觉语言

### 3.1 核心气质

- 保留“水墨流”作为审美方向，但从“强概念叙述”收口为“可执行视觉系统”。
- 视觉关键词固定为：`留白`、`纸面`、`墨色`、`秩序`、`轻层级`。
- 工作台不是复古装饰品。动画、纹理、拟物都只服务信息分层；画布层允许叠加极淡静态宣纸噪点，但不做动态噪点背景。

### 3.2 色彩 token

| Token | 用途 | 建议值 |
|---|---|---|
| `--bg-canvas` | 全局画布底色 | `#F7F4EE` |
| `--bg-surface` | 卡片与侧板 | `#FFFDF8` |
| `--bg-muted` | 次级分区 | `#F1ECE3` |
| `--text-primary` | 主文字 | `#1F1B16` |
| `--text-secondary` | 次级文字 | `#655C52` |
| `--accent-ink` | 强调色 | `#2E6F6A` |
| `--accent-info` | 信息状态 | `#3A7CA5` |
| `--accent-warning` | 警示状态 | `#B7791F` |
| `--accent-danger` | 危险状态 | `#B2412E` |
| `--accent-success` | 成功状态 | `#2F6B45` |

### 3.3 字体与排版

| 场景 | 建议字体 |
|---|---|
| 页面标题、章标题 | `Source Han Serif SC`, `Noto Serif SC`, serif |
| 正文、说明文字 | `Source Han Sans SC`, `Noto Sans SC`, sans-serif |
| YAML、Prompt、日志 | `JetBrains Mono`, `Cascadia Code`, monospace |

排版规则：

- 标题使用宋体系统建立作品感，正文说明使用黑体系统保证长时间阅读。
- 正文编辑区采用较宽行距和稳定字距，避免过度装饰。
- 所有状态信息都必须有文字标签，不能只靠颜色。
- 写作体验优先级固定为：首行缩进、阅读行宽、行间距、字重稳定性，高于锐化类装饰技巧。

### 3.4 动效原则

| 场景 | 规则 |
|---|---|
| 页面切换 | 用淡入淡出或轻位移，避免夸张墨迹覆盖动画 |
| `Version Panel` / 详情侧板 | 可用轻量折扇式开合，单次完成于 180ms 到 240ms，只使用 `transform/opacity` |
| Engine 节点状态 | `running` 用湿墨晕层，`completed` 用入纸收锋，避免工业呼吸灯 |
| 状态刷新 | 骨架屏优先，避免整页抖动 |
| `Ink Pool` 响应 | 波纹只绑定 token/账单更新，不跟随每条日志闪烁 |
| 长连接事件 | 仅高亮新增日志，不持续闪烁 |

补充：

- 所有动效都必须支持 `prefers-reduced-motion`，且 `Engine` 里的实时状态优先级高于装饰动效。
- 折扇式侧板只发生在面板开合，不延伸到 tab 切换和常规列表刷新。

### 3.5 水墨流组件规范

#### 3.5.1 枯墨与润墨的状态隐喻

**润墨（Active）状态**：

- 适用场景：正在编辑或刚生成的内容
- 视觉表现：边缘有极轻微的"湿润感"，通过 `text-shadow: 0 0 2px rgba(46, 111, 106, 0.15)` 实现极淡的晕染

**枯墨（Stale）状态**：

- 适用场景：失效内容，需要重新生成
- 视觉表现：透明度降低 + "枯笔"纹理效果，暗示内容已"干涸"
- 左侧边缘出现一条"干墨笔触"的垂直警示线

| 状态 | 视觉隐喻 | 用户感知 |
|---|---|---|
| Active（润墨） | 边缘晕染、微湿润感 | 新鲜、可继续编辑 |
| Stale（枯墨） | 干涸纹理、透明度降低 | 过时、需重新生成 |

#### 3.5.2 朱砂印章（验证反馈）

- 验证通过后显示"朱砂印章"图标，颜色使用 `--accent-success` 变体
- 带轻微的印压纹理感（`box-shadow: inset 0 0 4px rgba(0,0,0,0.3)`）
- 验证失败时，输入框底部出现"红墨泼溅"般的警示线条（1px 不规则波浪线）

#### 3.5.3 长卷滚动条

- 适用场景：Audit Log、Engine 日志流、Config Registry 列表
- 滚动条轨道：设计为一根"细长笔杆"，颜色使用 `--bg-muted`
- 滚动条滑块：设计为"笔头"，颜色使用 `--accent-ink`
- 蘸墨效果：滑块呈现渐变色（顶部浅、底部深），模拟蘸墨效果

#### 3.5.4 无障碍降级

```css
@media (prefers-reduced-motion: reduce) {
  .ink-ripple,
  .fan-open {
    transition: none !important;
    animation: none !important;
  }
  .seal-stamp {
    transform: none !important;
  }
}
```

## 4. 主视图设计

### 4.1 Lobby

**空间角色：** 像进入私人藏书馆，安静、清晰、项目优先。

**MVP 范围：**

- 项目卡片列表
- 新建项目入口
- 最近更新时间、项目状态、目标字数等摘要信息
- 回收站入口与已删除项目恢复
- 全局设置与凭证中心入口
- 项目设置侧栏到当前项目凭证中心的跳转入口

**Future：**

- 作品封面生成
- 项目分组与收藏体系
- 高级数据看板入口

### 4.2 Studio

**空间角色：** 以正文与前置资产为中心的写作桌面，强调专注和连续操作。

**MVP 布局：**

- 左侧：资产与章节导航
- 中央：设定编辑 / 资产编辑 / 章节正文编辑
- 右侧：版本面板或辅助信息侧板

**MVP 范围：**

- `ProjectSetting` 编辑与完整度检查
- `Outline` 草稿保存与确认
- `OpeningPlan` 草稿保存与确认
- 章节列表、章节详情、章节编辑、章节确认
- 章节版本列表、版本回滚、最佳版本标记
- `stale` 内容与下游影响的显式警示，默认使用墨迹褪色与警示标记

**MVP UI 占位：**

- AI 辅助编辑入口
- Story Bible 胶囊
- 跳转到当前 workflow 的 Review 面板

**Future：**

- 光标处 `/` 命令条
- 墨迹气泡式局部改写
- 透光叠层式版本对比与可视化批注

### 4.3 Engine

**空间角色：** 不是流程画布秀场，而是执行控制台。优先级是“当前工作流到底在做什么”。

**MVP 布局：**

- 顶部：工作流状态、模式、主控制按钮
- 左侧：章节任务列表
- 中央：执行概览与节点时间线
- 右侧：日志、审核、账单、上下文预览、Prompt 回放

**MVP 范围：**

- 启动、暂停、恢复、取消工作流
- 查看当前 `manual / auto` 模式
- 重建章节任务与编辑任务草稿
- 查看节点执行、执行日志、审核摘要、账单摘要、上下文预览、Prompt 回放
- 从工作流发起导出

**MVP UI 占位：**

- 节点卡片正反翻转
- 节点参数可视化详情

**Future：**

- 运行时切换手动/自动模式
- Dry-run 成本预估
- Workflow / Skill Schema 校验弹窗
- Prompt Lab、模拟执行、保存为 Skill

### 4.4 Lab

**空间角色：** 偏研究工作台，而不是“上传小说后一键炼丹”的实验场。

**MVP 布局：**

- 左侧：分析记录列表和过滤器
- 中央：分析详情
- 右侧：新建分析或结果摘要

**MVP 范围：**

- 按项目查看分析结果
- 按 `analysis_type` / `content_id` / `generated_skill_key` 过滤
- 新建分析记录，显式写入 `analysis_type` / `source_title` / `generated_skill_key` / `result` / `suggestions`
- 查看分析详情与 `generated_skill_key`
- 在详情区删除当前分析记录

**MVP UI 占位：**

- “来自某个章节或某次导入”的来源说明

**Future：**

- 原文上传
- 文本采样
- 自动分析流水线
- 试写与保存 Skill

## 5. 设计收口规则

### 5.1 顶层页面不再继续膨胀

- `Review & Diff` 不是新的一级页面。
- `Export` 不是新的一级页面。
- `Recycle Bin` 不是新的一级页面。
- `Version History` 不是新的一级页面。

### 5.2 可视化必须服从数据真值

- UI 文案不能把未来态能力写成当前能力。
- 原始后端状态可以复杂，但展示文案必须统一、稳定、可理解。
- 未落地能力如需保留入口，必须标明 `即将支持` 或使用禁用态。
- `MVP UI 占位` 只允许做说明性预留，不占用主 CTA，不伪装成可立即执行的功能。

### 5.3 破坏性操作必须克制

- `Delete Project` 进入回收站，不在 MVP 主流程中暴露永久删除。
- `Export` 只展示当前支持格式，不做“先画入口后报错”的假能力。
- `Workflow` 的高级实验开关不先行暴露。

## 6. 实施前检查项

- 主视图是否仍稳定为 `Lobby / Studio / Engine / Lab`
- `Auth` 是否保持独立入口而不是塞进工作台内部
- `Studio` 是否已围绕 `ProjectSetting -> Outline -> OpeningPlan -> Chapter` 展开
- `Engine` 是否优先展示状态、任务和执行证据，而不是未来编排器幻想
- `Lab` 是否已收口为“分析结果工作台”而非“上传采样一体机”
- 所有 `Future` 能力是否都已明确标注，不再伪装成“已完善”
